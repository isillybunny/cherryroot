Block 0 · PC Basics
Before Everything
Antes de hackear qualquer coisa, você precisa entender o que é um computador. Este bloco é a fundação de todo o resto.
HardwareCPU & RAMOS File SystemTerminalPermissionsPorts
14 terms · B·01 → B·14 start here
HardwareB·01
Tudo que você pode tocar. Teclado, mouse, CPU, RAM, disco. A base física de tudo.
Sem hardware, nada existe. Acesso físico = acesso total.
CPUB·02
O cérebro. Busca instrução na RAM → decodifica → executa → repete bilhões de vezes/segundo.
Exploits de memória colocam código malicioso onde a CPU vai encontrar e executar.
RAMB·03
Memória temporária e rápida. Some quando desliga. Pode conter senhas e chaves de criptografia.
RAM = volátil. Disco = persistente. Diferença crítica em forense.
SoftwareB·04
hardware o que fazer. Sem forma física. Malware é software — só a intenção muda.
A mecânica entre programa legítimo e malware é idêntica.
Operating SystemB·05
hardware e apps. Linux domina em segurança — a maioria dos alvos roda nele.
Quem controla o OS controla tudo que roda nele.
Binary & HexB·06
Sistemas numéricos reais do computador. Binary = 0s e 1s. Hex = base 16. `41` hex = `A` em texto.
Hex aparece em todo lugar em segurança. Aprenda a reconhecer os padrões.
File SystemB·07
Organiza tudo a partir de `/`. Saber: `/etc/shadow` = hashes · `/var/log` = logs · `/tmp` = world-writable.
No Linux tudo é arquivo — até dispositivos e processos.
PathB·08
Endereço de um arquivo. Absoluto = `/etc/passwd`. Relativo = `../config`. `../` em URL = path traversal.
path traversal imediato.
PermissionsB·09
Controla quem lê, escreve ou executa. `rwxr-xr--`. SUID owned by root = caminho pra escalada.
`chmod 777` = todo mundo pode tudo. Nunca em arquivos sensíveis.
User & AdminB·10
Usuário = acesso limitado. Root (UID 0) = zero restrições. `sudo -l` mostra seu caminho pro root.
Conseguir root = objetivo de todo ataque. Proteger root = objetivo de toda defesa.
Terminal / CLIB·11
Interface de texto. Você digita → shell interpreta → OS executa. Toda ferramenta de segurança vive aqui.
"Popar uma shell" num alvo = execução remota de código = controle total.
Env VariablesB·12
Valores globais que qualquer processo pode ler. PATH, USER, HOME. API keys em ENV = perigo.
Controlar PATH = controlar o que roda quando você digita qualquer comando.
ProcessB·13
Programa em execução com seu PID e espaço de memória. Malware é um processo. `ps aux` mostra todos.
Processo desconhecido rodando como root = red flag. Investigue imediatamente.
PortB·14
Canal numerado que um serviço escuta. 22=SSH · 80=HTTP · 443=HTTPS. Porta aberta = entrada potencial.
Nmap encontra portas abertas. Se está aberta, é um alvo em potencial.
BIOS / UEFIB·15
Firmware que inicializa hardware antes do OS. Configura boot order, senhas, secure boot.
BIOS comprometida = persistÃªncia abaixo do OS. Quase impossÃ­vel remover.
KernelB·16
NÃºcleo do OS que gerencia hardware, processos, memÃ³ria. Bug no kernel = jackpot.
Kernel exploit = privilÃ©gios máximos. Toda defesa do OS cai junto.
SyscallB·17
Ponte entre app userspace e kernel. `open()`, `read()`, `execve()` são syscalls. Strace mostra todas.
Monitorar syscalls = ver tudo o que um programa realmente faz no sistema.
Processos vs ThreadsB·18
Processo tem memória própria; threads compartilham memória do processo pai.
Thread injection = injetar código dentro de um processo legitimo já rodando.
Swap MemoryB·19
Disco usado como extensão da RAM quando lotada. Pode vazar dados sensíveis após reboot.
Senhas e chaves podem parar no swap. Forênsica adora analisar isso.
Boot ProcessB·20
POST → BIOS/UEFI → Bootloader (GRUB) → Kernel → Init → User Space.
Cada etapa do boot é um ponto onde malware pode se instalar (bootkit).
Registry (Windows)B·21
Banco de config global do Windows. HKLM, HKCU. Malware adora `Run` keys.
`HKLM\Software\Microsoft\Windows\CurrentVersion\Run` = autorun na inicialização.
DriversB·22
Software que faz hardware funcionar. Drivers com bug = kernel exploits (BYOVD).
BYOVD = Bring Your Own Vulnerable Driver. Técnica popular de atacantes em 2024+.

Block 1 · How the Internet Works
The Internet
Tudo em web security acontece pela internet. Este bloco explica como dados viajam e o que acontece toda vez que você abre um navegador.
IP AddressDNSHTTP/HTTPS CookiesSessionsAPISSL/TLS
14 termos · W·01 → W·14 how the web works
Internet vs WebW·01
"Internet = a infraestrutura. Web = um serviço que roda em cima dela."
Internet = a rede global de computadores. Web = o serviço que você acessa pelo navegador usando HTTP.
internetwebHTTPbrowserprotocol
Email, SSH e games também usam a internet — mas não são a web. Não confunda os dois.
IP AddressW·02
"O endereço único de um dispositivo na rede — como um CEP para computadores."
IPv4 = 4 números (192.168.0.1). IPv6 = mais longo. IP privado = dentro da rede. IP público = visível na internet.
IPv4IPv6private IPpublic IPNATlocalhost
Attack
  • IP scanning pra mapear alvos
  • IP spoofing pra falsificar origem
Defense
  • VPN mascara seu IP público
  • Firewall bloqueia IPs suspeitos
Seu IP público é visível pra todo site que você visita. VPN troca pelo IP do servidor VPN.
DNSW·03
"A agenda telefônica da internet — traduz nomes de domínio em IPs."
Você digita google.com → DNS resolver consulta a hierarquia → retorna o IP → browser conecta.
DNS resolverA recordMX recordCNAMETTLnameserver
Attack
  • DNS spoofing — injetar respostas falsas
  • DNS cache poisoning
  • Subdomain takeover
Defense
  • DNSSEC autentica os registros
  • Auditoria de subdomínios
DNS é o primeiro lugar que um hacker investiga. Registros DNS revelam muito sobre a infraestrutura do alvo.
HTTP / HTTPSW·04
"O protocolo que carrega toda página web que você já viu."
HTTP = sem criptografia. HTTPS = HTTP dentro de um túnel TLS. Métodos: GET (busca), POST (envia), PUT (atualiza), DELETE (remove).
GETPOSTTLSstatelessport 80port 443
Attack
  • MITM em HTTP não criptografado
  • SSL stripping — força downgrade
Defense
  • Sempre HTTPS — nunca HTTP
  • HSTS força HTTPS no browser
HTTP é a camada onde quase todos os ataques web acontecem. Entenda requests e responses de cor.
URLW·05
"O endereço completo de qualquer recurso na web."
`https://site.com/page?id=1#section` → protocol / domain / path / query string / fragment
domainpathquery stringparameterfragmentsubdomain
Attack
  • Injetar payloads nos parâmetros
  • Path traversal via path
  • Phishing com domínios similares
Defense
  • Validar e sanitizar parâmetros
  • Usar allowlists de valores esperados
O query string é onde a maioria dos ataques começa. Trate todo parâmetro de URL como input não confiável.
Request & ResponseW·06
"Browser pergunta (request). Servidor responde (response). Todo ciclo na web segue isso."
Request: method + path + headers + body. Response: status code + headers + body (a página).
requestresponseheadersbodystatus codeBurp Suite
Attack
  • Interceptar e modificar requests
  • Replay de requests válidos
  • Fuzzing de parâmetros
Defense
  • Validar toda entrada server-side
  • CSRF tokens anti-replay
Se você consegue interceptar e modificar requests, consegue testar quase toda vulnerabilidade web.
Status CodesW·07
"O código numérico que o servidor retorna dizendo o que aconteceu."
2xx = sucesso · 3xx = redirect · 4xx = erro do cliente · 5xx = erro do servidor
200 OK301 redirect401 unauth403 forbidden404 not found500 error
Attack
  • 403 = recurso existe, tente bypass
  • 500 em injection = possível vuln
  • Códigos diferentes = enumeração
Defense
  • Mensagens de erro genéricas
  • Não expor detalhes internos em 500
Status codes são pistas. Um hacker lê cada código como um mapa do que existe no servidor.
HeadersW·08
"Metadados enviados junto com todo request e response."
Request: Host, User-Agent, Cookie, Authorization. Response: Set-Cookie, Content-Type, CSP, HSTS, X-Frame-Options.
HostCookieAuthorizationCSPHSTSX-Frame-Options
Attack
  • Host header injection
  • Headers faltando = XSS / clickjacking
  • Server header vaza versão do software
Defense
  • Adicionar: CSP, HSTS, X-Frame-Options
  • Remover headers que revelam versões
Headers de segurança faltando são um achado em quase todo pentest. Sempre cheque o que o servidor retorna.
CookiesW·09
"Pequenos dados que o servidor armazena no browser para lembrar o usuário."
HTTP é stateless — cookies resolvem isso. Login → servidor envia cookie com session ID → browser manda em todo request.
session IDHttpOnlySecure flagSameSitecookie theft
Attack
  • XSS rouba session cookie = account takeover
  • Session fixation — forçar session ID conhecido
Defense
  • HttpOnly — JS não lê o cookie
  • Secure — só enviado via HTTPS
  • SameSite=Strict — previne CSRF
session cookie = roubar a sessão logada. É por isso que XSS é tão perigoso.
SessionsW·10
"Registro server-side que mantém o estado do usuário logado."
Login → session criada no servidor → ID enviado via cookie → cada request verifica o ID → logout destrói a session.
session IDsession hijackingsession fixationsession expirystateful
Attack
  • Hijacking — roubar ou prever session ID
  • Sessions que nunca expiram
Defense
  • Regenerar session ID após login
  • Timeout de inatividade
  • Invalidar completamente no logout
O session ID é a coroa. Proteja como uma senha — porque é exatamente isso.
APIW·11
"Interface que deixa programas se comunicarem — geralmente via HTTP e JSON."
App envia GET /endpoint → API retorna JSON → app exibe. REST APIs usam métodos HTTP pra interagir com recursos.
RESTendpointJSONAPI keyrate limitingIDOR
Attack
  • Endpoints sem autenticação
  • IDOR via referências de objetos
  • API keys expostas no código
Defense
  • Autenticar todo endpoint
  • Rate limiting e throttling
  • Nunca expor API keys client-side
APIs são a nova superfície de ataque. Sempre teste a API ao testar uma aplicação web.
Client & ServerW·12
"Cliente = quem pede. Servidor = quem responde."
Código client-side (HTML, JS) pode sempre ser visto e modificado pelo usuário. Validação real só existe no servidor.
client-sideserver-sidebrowserweb servervalidation
Attack
  • Bypass de validação client-side
  • Modificar JS pra remover restrições
Defense
  • Validação sempre server-side
  • Nunca confiar em dados do cliente
Qualquer coisa que o cliente envia pode ser forjada. Segurança real só existe no servidor.
CDNW·13
"Rede de servidores distribuídos que entrega conteúdo do ponto mais próximo do usuário."
CDN cacheia conteúdo em edge servers ao redor do mundo. Também pode atuar como WAF filtrando tráfego malicioso.
Cloudflareedge serverorigin servercachingWAF
Attack
  • Achar IP do origin bypassa CDN e WAF
  • Cache poisoning — injetar conteúdo
Defense
  • Nunca expor IP do origin server
  • Regras WAF no CDN
Encontrar o IP do servidor de origem bypassa o CDN e toda a WAF que o protege.
SSL / TLSW·14
"O protocolo criptográfico que faz o HTTPS funcionar."
TLS garante: encriptação (dados ilegíveis em trânsito) + autenticação (servidor prova identidade) + integridade (dados não foram alterados).
TLScertificateCAhandshakecipher suiteHSTS
Attack
  • SSL stripping — forçar downgrade pra HTTP
  • Cipher suites fracos — craquear criptografia
Defense
  • TLS 1.2 ou 1.3 no mínimo
  • HSTS previne SSL stripping
  • Certificados de CAs confiáveis
O cadeado = conexão criptografada. Não significa que o site é seguro ou legítimo.
WebSocketW·15
Conexão persistente bidirecional cliente-servidor. Usado em chat, real-time. Não respeita CORS.
Burp e ZAP capturam mensagens WebSocket — teste injection nelas também.
GraphQLW·16
Alternativa a REST. Cliente pede só os campos que quer. Vulnerável a introspecção mal protegida e batching attacks.
Endpoint `/graphql` com introspecção aberta = mapa completo da API entregue.
OAuth 2.0W·17
Framework de autorização delegada. Login com Google/GitHub. Redirect_uri mal validado = takeover.
`redirect_uri` sem allowlist estrita = vazamento de código de autorização.
JWT (JSON Web Token)W·18
Token assinado JSON usado pra auth stateless. Atenção: `alg:none`, chave fraca, secrets em URL.
JWT em URL aparece em logs e em `Referer`. Sempre transporte em header ou cookie.
CORSW·19
Cross-Origin Resource Sharing. Controla quem chama sua API de qual origem. Wildcard `*` + credentials = perigo.
`Access-Control-Allow-Origin: *` com cookies = qualquer site rouba sessão.
CSP (Content Security Policy)W·20
Header que mitiga XSS. Define o que browser pode carregar/executar. `unsafe-inline` quebra.
Boa CSP força o atacante a achar bypass criativo. Sem ela, qualquer XSS é trivial.
HSTSW·21
Header que força HTTPS. Browser memoriza por X segundos. Mitiga SSL strip.
`Strict-Transport-Security` com `preload` = browser nunca tenta HTTP, nem na primeira vez.
Same-Site CookieW·22
Atributo de cookie que controla envio em requests cross-site. `Strict`, `Lax`, `None`. Defesa contra CSRF.
`SameSite=Strict` em cookies de sessão quebra a maioria dos ataques CSRF de graça.

Block 2 · Networks
The Network
Todo ataque viaja por uma rede. Este bloco explica como redes são estruturadas, como dispositivos se comunicam e como atacantes se movem por elas.
OSI ModelTCP/IPFirewall RouterVPNDHCPSubnet
17 terms · N·01 → N·17 how attacks travel
NetworkN·01
Grupo de dispositivos conectados que podem se comunicar. Do Wi-Fi de casa à internet inteira.
LANWANdevicescomunicação
Attack
  • Sniffing de tráfego não criptografado
  • Lateral movement entre dispositivos
Defense
  • Segmentação de rede
  • Criptografia em todas as conexões
Todo ataque viaja por uma rede. Entender redes = entender como ataques se movem.
LAN / WANN·02
LAN = rede local (casa, escritório). WAN = longa distância (internet). Dentro da LAN, dispositivos se comunicam diretamente.
ethernetwifiinternetroteamento
Attack
  • Atacante dentro da LAN tem visão total
  • WAN exposta = qualquer um pode tentar
Defense
  • Firewall entre LAN e WAN
  • Subnetting + zero trust interno
Estar dentro da LAN é perigoso — redes internas confiam em tudo dentro delas por padrão.
OSI ModelN·03
7 camadas: Physical → Data Link → Network → Transport → Session → Presentation → Application. Cada camada tem seus próprios ataques.
7 camadasphysicaltransportapplication
Attack
  • Cada camada tem ataques específicos
  • L2 = ARP spoof · L7 = SQL injection
Defense
  • Defesa em profundidade — múltiplas camadas
  • Conhecer onde cada ataque acontece
L3 = Network (IP). L7 = Application (HTTP, ataques web). "Layer X attack" = ataque naquela camada.
TCP / IPN·04
TCP garante entrega. IP cuida do endereçamento. Three-way handshake: SYN → SYN-ACK → ACK = conexão estabelecida.
handshakeSYNACKreliable
Attack
  • SYN flood — exaurir recursos do servidor
  • Sequence prediction (raro hoje)
Defense
  • SYN cookies habilitados
  • Rate limiting de conexões
SYN flood = enviar milhões de SYN sem completar o handshake = servidor esgotado (DoS).
UDPN·05
Mais rápido que TCP mas sem confirmação de entrega. Usado em DNS, games, VoIP, streaming.
fastno-ackDNSstreaming
Attack
  • UDP amplification — DNS, NTP, memcached
  • Spoofing fácil sem handshake
Defense
  • Rate limiting nos servidores UDP
  • Filtros anti-spoof no roteador
UDP amplification = pequeno request dispara resposta enorme = flood no alvo.
MAC AddressN·06
Endereço físico permanente de cada placa de rede. Usado dentro da LAN. IP = internet, MAC = rede local.
layer 2físicoNICpermanente
Attack
  • ARP spoofing — interceptar tráfego local
  • MAC flooding em switches
Defense
  • Port security em switches
  • Static ARP em servidores críticos
ARP spoofing = ligar seu MAC ao IP de outra pessoa = você recebe o tráfego dela.
RouterN·07
Conecta redes diferentes e decide pra onde cada pacote vai. Seu gateway pra internet.
gatewayroutingNATborder
Attack
  • Comprometer router = ver tudo que passa
  • Firmware desatualizado tem CVEs
Defense
  • Atualização constante de firmware
  • Senhas fortes + admin não exposto
Comprometer o router = ver todo o tráfego que passa por ele. É o gatekeeper da rede.
SwitchN·08
Conecta dispositivos dentro da mesma rede. Aprende quem está em qual porta e direciona tráfego.
layer 2CAM tableMACLAN
Attack
  • MAC flooding → switch vira hub → sniffing
  • VLAN hopping em configs fracas
Defense
  • Port security limitando MACs
  • VLAN configurada corretamente
MAC flooding enche a tabela → switch vira hub → transmite tudo pra todo mundo → sniffing fácil.
FirewallN·09
Filtra tráfego por regras. Permite ou bloqueia pacotes. Verifica IP, porta, protocolo. Primeira linha de defesa.
rulesfilterallowblock
Attack
  • Bypass via porta permitida
  • Tunneling dentro de tráfego legítimo
Defense
  • Default deny — bloquear por padrão
  • Auditoria regular de regras
Firewall é tão forte quanto suas regras. Regra mal configurada = porta aberta sem você saber.
Ports & ProtocolsN·10
Portas = as portas. Protocolos = o idioma falado por elas. 22=SSH · 80=HTTP · 443=HTTPS · 3389=RDP.
TCPUDP2280443
Attack
  • Port scan revela serviços rodando
  • Cada porta aberta é entrada potencial
Defense
  • Fechar portas não usadas
  • Firewall + monitoramento de scan
Porta aberta = serviço rodando = potencial vulnerabilidade. Nmap acha todas.
PacketN·11
Pequena unidade de dados que viaja pela rede. Header (metadados) + payload (dados). TCP reassembla na ordem certa.
headerpayloadfragmentopacote
Attack
  • Sniffing de pacotes não criptografados
  • Packet injection / replay
Defense
  • Sempre HTTPS / TLS
  • IDS para detectar pacotes anômalos
Tráfego não criptografado = qualquer um que capture os pacotes consegue ler seus dados.
BandwidthN·12
Capacidade máxima de dados por segundo. DDoS = saturar o bandwidth do alvo até ficar inacessível.
throughputmbpscapacitybottleneck
Attack
  • DDoS satura o bandwidth do alvo
  • Volumetric attacks até cair
Defense
  • CDN + scrubbing services
  • Anti-DDoS na borda da rede
Bandwidth = largura do cano. DDoS entope o cano até ninguém mais passar.
ProxyN·13
Intermediário entre cliente e servidor. Forward = frente do cliente. Reverse = frente do servidor. Burp Suite é um proxy.
intermediárioforwardreversecache
Attack
  • Man-in-the-middle via proxy malicioso
  • Burp intercepta tráfego web
Defense
  • Certificate pinning em apps
  • TLS forte impede leitura via proxy
Burp Suite intercepta tudo entre seu browser e o alvo — é assim que você testa web apps.
VPNN·14
Túnel criptografado que esconde seu IP real e protege seu tráfego do ISP.
tunnelencryptedIP maskremote
Attack
  • VPN comprometida = acesso direto à rede
  • Provider VPN vê tudo
Defense
  • VPN corporativa com MFA
  • VPN confiável só pra anonimato pessoal
VPN esconde seu IP de sites mas não do provedor VPN. Não é magia — é só um túnel.
NATN·15
Traduz IPs privados pra um IP público. Permite que vários dispositivos compartilhem um IP.
translationprivate IPpublic IP
Attack
  • NAT não é segurança real — só obscurece
  • Bypass via UPnP em redes domésticas
Defense
  • Firewall de verdade, não confie no NAT
  • Desabilitar UPnP se não usado
NAT é conveniência, não segurança. Use firewall de verdade — não confie no NAT.
SubnetN·16
Divisão de uma rede maior em segmentos menores. /24 = 256 endereços. Segmentação limita movimento de atacantes.
CIDR/24segmentationbroadcast
Attack
  • Sem segmentação = breach total
  • Lateral movement irrestrito
Defense
  • Microsegmentação de rede
  • VLANs separando departamentos
Boa segmentação = breach numa subnet não atinge automaticamente as outras.
DHCPN·17
Atribui IPs automaticamente. DORA: Discover → Offer → Request → Acknowledge.
IP atribuídoDORAleasegateway
Attack
  • Rogue DHCP server = redirecionar tráfego
  • DHCP starvation esgota IPs
Defense
  • DHCP snooping ativado
  • Logs de leases monitorados
Rogue DHCP server distribui gateway falso = todo tráfego da rede passa pelo atacante.
ARPN·18
Resolve IP em MAC na rede local. ARP spoofing redireciona tráfego (MITM).
ARP não tem autenticação. Quem responde primeiro vence. Ótimo pra MITM em LAN.
ICMPN·19
Protocolo de mensagens de rede. Ping, traceroute usam. Tunneling em ICMP exfiltra dados.
Firewalls costumam liberar ICMP. Atacantes escondem C2 em pacotes ping.
BGPN·20
Protocolo de roteamento entre AS. Hijack pode redirecionar grandes ranges de IPs.

Red Flags
Sinais que algo está errado — ou que vale investigar imediatamente.
Numa URL ou Request
../ no parâmetro de URL — tentativa de path traversal, teste imediato
?id=1 na URL — teste IDOR (id=2, id=0, id=-1)
' ou " em campos de input — possível SQL injection
Input refletido na página — possível XSS
url= ou redirect= — teste SSRF e open redirect
file= ou path= — teste path traversal e LFI
Numa resposta do servidor
500 com input injetado — servidor quebrou, injeção pode ter funcionado
Stack trace na página de erro — vaza tecnologia, paths, info interna
403 num recurso — existe mas bloqueado, tente bypass
Headers de segurança ausentes — sem CSP, HSTS, X-Frame-Options
Server header exposto — Apache/2.4.1 vaza versão
Cookie sem HttpOnly — JS pode ler, risco de XSS
Num sistema Linux
Arquivo SUID owned by root — executa como root, cheque GTFOBins
Processo desconhecido como root — pode ser malware
Diretório world-writable — qualquer um escreve, cheque arquivos plantados
sudo -l mostra (ALL) — pode rodar tudo como root
Cron rodando como root com script gravável — root shell garantido
auth.log apagado — alguém apagou rastros, breach provável
Num scan de rede
Porta 3389 aberta (RDP) — Windows remoto, teste credenciais padrão
Porta 445 aberta (SMB) — cheque EternalBlue e null sessions
Credenciais padrão funcionando — admin/admin, admin/password
FTP permite anônimo — qualquer um loga, cheque arquivos sensíveis
No mundo real — phishing, fraude e engenharia social
!Typosquatting na URL
Domínios parecidos como paypa1.com em vez de paypal.com, ou g00gle.com. Confira letra por letra antes de digitar credenciais.
!Pop-up urgente
"Seu computador está infectado! Ligue agora!" — pânico é a tática. Empresas reais não avisam infecção por pop-up no browser.
!Anexo .exe ou .scr
Executáveis vindos por email são quase sempre malware. Mesmo fatura.pdf.exe — o Windows esconde a extensão real.
!Pedido de senha por email
Nenhum banco, empresa ou serviço sério pede senha por email. Se pediu, é phishing — não responda, denuncie.
!HTTPS ausente no login
Formulário de login sem cadeado = senha trafega em texto puro. Qualquer um na rede consegue ler.
!Permissões excessivas em apps
Lanterninha que pede contatos, microfone e localização? Provavelmente vende seus dados — ou é malware disfarçado.
!Encurtador de URL em email
bit.ly, t.co, tinyurl em email institucional escondem o destino real. Expanda antes de clicar (unshorten.it).
!Telefonema "do banco"
Ligações pedindo dados, token ou senha por telefone — desligue. Ligue de volta no número oficial do cartão.
!QR code em lugar público
Adesivos com QR em parquímetros, cardápios, postes — podem ter sido sobrepostos. Cheque o domínio antes de pagar.
!Wi-Fi público sem senha
Qualquer um na rede pode interceptar tráfego. Use VPN, ou só acesse HTTPS. Nunca faça login bancário.
!Pendrive encontrado
USB drops são técnica clássica: atacante "perde" pendrive na empresa esperando alguém plugar. Pode ser malware ou BadUSB.
!"Click here to fix"
Mensagem de erro com botão pra "consertar" — quase sempre instala adware ou abre porta pro atacante. Feche a aba.

Códigos HTTP
Status codes essenciais — o que cada faixa significa e o que olhar em pentest.
1xx · Informacional
100Continue — servidor recebeu cabeçalhos, pode mandar o corpo
101Switching Protocols — usado em upgrade pra WebSocket
2xx · Sucesso
200OK — requisição bem-sucedida, resposta normal
201Created — recurso criado (POST que deu certo)
204No Content — sucesso sem corpo na resposta
206Partial Content — resposta parcial (range requests)
3xx · Redirecionamento
301Moved Permanently — recurso mudou de URL pra sempre
302Found — redirect temporário, comum após login
304Not Modified — cache do cliente ainda é válido
307Temporary Redirect — preserva método HTTP original
4xx · Erro do cliente
400Bad Request — requisição mal formada
401Unauthorized — falta autenticação válida
403Forbidden — autenticado mas sem permissão. Tente bypass
404Not Found — recurso não existe (ou está escondido)
405Method Not Allowed — verbo HTTP errado pra esse endpoint
418I'm a teapot — easter egg, não usar a sério
429Too Many Requests — rate limit ativado
5xx · Erro do servidor
500Internal Server Error — algo quebrou. Pode ser injeção
502Bad Gateway — proxy reverso não conseguiu falar com backend
503Service Unavailable — servidor sobrecarregado ou em manutenção
504Gateway Timeout — backend demorou demais pra responder
Em pentest: 401 vs 403 importa — 401 pede credencial, 403 já tem mas falta permissão. 500 com payload injetado é frequentemente sinal de injeção funcionando.

Atalhos Terminal
Atalhos essenciais pra ganhar velocidade no bash, zsh e shells em geral.
Controle do processo
Ctrl+CInterrompe o comando atual (SIGINT)
Ctrl+DFecha o shell (EOF) — sai do terminal
Ctrl+ZPausa o processo em background (fg pra retomar)
Ctrl+LLimpa a tela (igual ao comando clear)
Edição da linha
Ctrl+AVai pro início da linha
Ctrl+EVai pro fim da linha
Ctrl+UApaga tudo do cursor até o início
Ctrl+KApaga tudo do cursor até o fim
Ctrl+WApaga a palavra anterior
Ctrl+YCola o último texto apagado (yank)
Alt+B / Alt+FPula palavra pra trás / pra frente
Histórico
↑ / ↓Navega pelo histórico de comandos
Ctrl+RBusca reversa no histórico — digite parte do comando
!!Repete o último comando
!$Último argumento do comando anterior
!*Todos os argumentos do comando anterior
!sudoÚltimo comando que começou com "sudo"
sudo !!Repete o último comando com sudo
Completar e navegar
TabAutocompletar arquivo, pasta ou comando
Tab TabLista todas as opções possíveis de completação
cd -Volta pro diretório anterior
cd ~Vai pro diretório home
Dominar Ctrl+R muda a vida — não precisa lembrar comando inteiro, só um pedaço.

Regex Básico
Regex de uma página — metacaracteres, classes, quantificadores e exemplos práticos.
Metacaracteres essenciais
.Qualquer caractere (exceto quebra de linha)
^Início da linha
$Fim da linha
\Escapa o próximo caractere — \. casa um ponto literal
|OU lógico — gato|cachorro casa um ou outro
Quantificadores
*Zero ou mais — guloso (pega o máximo possível)
+Um ou mais
?Zero ou um (opcional) — também torna o quantificador anterior preguiçoso
{n}Exatamente n vezes
{n,m}Entre n e m vezes
{n,}No mínimo n vezes
Classes de caracteres
[abc]Qualquer um entre a, b ou c
[^abc]Qualquer um EXCETO a, b ou c
[a-z]Faixa — qualquer letra minúscula
\dDígito (0-9)
\DNão-dígito
\wCaractere de palavra (letras, dígitos, _)
\WNão-palavra
\sEspaço, tab ou quebra de linha
\SNão-espaço
\bLimite de palavra
Grupos e referências
(abc)Grupo de captura — pode ser referenciado depois
(?:abc)Grupo sem captura — só agrupa, não guarda
\1Backreference ao primeiro grupo capturado
(?=abc)Lookahead positivo — "seguido de abc" sem consumir
(?!abc)Lookahead negativo — "não seguido de abc"
Exemplos práticos
^\d{3}\.\d{3}\.\d{3}-\d{2}$CPF no formato 000.000.000-00
\b[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}\bEmail básico
^https?://Linha que começa com http:// ou https://
(?i)senhaCasa "senha", "Senha", "SENHA" (case-insensitive)
\b\d{1,3}(\.\d{1,3}){3}\bIP v4 (sem validar faixa)
Regex é uma linguagem — toda ferramenta de segurança usa (grep, sed, awk, Burp, scanners). Pratique no regex101.com.
