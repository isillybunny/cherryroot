Glossário Pessoal
Definições em uma linha para ativar a memória rápido. Ordem alfabética. Todos os blocos cobertos.
A
ACLAccess Control List — regras que definem quem pode acessar qual recurso
algorithmConjunto de instruções passo a passo que um computador segue
APIApplication Programming Interface — permite que programas se comuniquem
ASCIICodificação padrão que mapeia caracteres de texto em números
asymmetric encryptionUsa uma chave pública para criptografar e uma chave privada para descriptografar
attack surfaceTudo que está exposto e pode ser alvo de um atacante
authenticationProvar quem você é — login, senha, biometria
authorizationO que você tem permissão de fazer após se autenticar
B
backdoorAcesso oculto deixado aberto em um sistema para entrada futura
baitingDeixar algo tentador (USB, link) para enganar uma vítima
bandwidthCapacidade máxima de transferência de dados de uma conexão
bashShell mais comum no Linux — interpreta seus comandos
binarySistema numérico de base 2 — apenas 0s e 1s
bitMenor unidade de dado — um único 0 ou 1
blue teamTime responsável pela defesa, monitoramento e resposta a incidentes
bootloaderSoftware que carrega o OS quando você liga o computador
botnetRede de máquinas infectadas sob controle de um atacante
broken authExplorar lógica fraca de login ou sessão para obter acesso
bufferÁrea de memória temporária para dados em trânsito
bug bountyPrograma que paga pesquisadores para encontrar vulnerabilidades reais
byte8 bits — menor unidade endereçável de memória
C
C2 serverCommand and Control — centro do atacante para gerenciar o botnet
CDNContent Delivery Network — servidores distribuídos que aceleram a entrega
certificateDocumento digital que prova identidade na web — usado no HTTPS
chmodComando Linux para alterar permissões de arquivos
CIA TriadConfidencialidade, Integridade, Disponibilidade — os três pilares da segurança
CLICommand Line Interface — interação com o sistema via comandos de texto
clientO dispositivo ou programa que envia uma requisição ao servidor
closed portUma porta que não aceita conexões — não está escutando
command injectionExecutar comandos do sistema por meio de entrada não sanitizada do usuário
cookiePequeno arquivo armazenado no navegador para lembrar o estado do usuário
core (CPU)Unidade de processamento individual dentro de uma CPU — mais cores = mais trabalho paralelo
covering tracksRemover logs e evidências de uma intrusão
CPUCentral Processing Unit — o cérebro que executa todas as instruções
CSRFCross-Site Request Forgery — enganar um usuário para que faça requisições indesejadas
CTFCapture The Flag — competição legal de hacking para praticar habilidades
CVECommon Vulnerabilities and Exposures — ID único para uma vulnerabilidade conhecida
cryptographyTransformar dados em forma ilegível para protegê-los de acesso não autorizado
D
DHCPDynamic Host Configuration Protocol — atribui endereços IP automaticamente
distroUma versão e sabor do Linux (ex.: Kali, Ubuntu, Debian)
DNSDomain Name System — traduz nomes de domínio em endereços IP
dropperMalware cujo objetivo é instalar outro malware
E
encryptionTransformar dados para que apenas partes autorizadas possam lê-los
env variableValor global armazenado em memória que qualquer programa pode ler
executableUm arquivo que o OS pode executar como programa (binário)
exploitCódigo ou técnica que abusa de uma vulnerabilidade para obter acesso
exploitationA fase de usar uma vulnerabilidade para de fato obter acesso
ext4Formato padrão de sistema de arquivos do Linux
F
FAT32Sistema de arquivos usado em pen drives e mídias de armazenamento antigas
file systemSistema que organiza e endereça todos os arquivos no disco
firewallFiltra o tráfego de rede usando regras — permite ou bloqueia pacotes
firmwareSoftware de baixo nível embutido diretamente no hardware
forensicsInvestigar o que aconteceu durante ou após um incidente de segurança
functionBloco reutilizável de código que realiza uma tarefa específica
G
grepComando Linux para buscar padrões dentro de arquivos ou saída
H
hardeningReduzir a superfície de ataque removendo fraquezas e serviços desnecessários
hardwarePartes físicas de um computador — teclado, CPU, RAM, disco
hashImpressão digital unidirecional de dados — não pode ser revertida
heapParte da RAM usada para alocação dinâmica de memória em tempo de execução
hexHexadecimal — sistema numérico de base 16, abreviação para binário
honeypotSistema falso projetado para atrair e estudar atacantes
HTTPHyperText Transfer Protocol — protocolo para carregar páginas web
HTTPSHTTP com criptografia TLS — a versão segura do HTTP
I
IDORInsecure Direct Object Reference — acessar dados de outro usuário alterando um ID
IDSIntrusion Detection System — monitora e alerta sobre atividade suspeita
impersonationFingir ser uma pessoa confiável para extrair informações ou acesso
incident responseProcesso estruturado para lidar com e se recuperar de uma violação de segurança
IPSIntrusion Prevention System — detecta e bloqueia ativamente atividade suspeita
IP addressEndereço numérico único de um dispositivo em uma rede
K
kernelNúcleo do OS — gerencia recursos de hardware diretamente
keyloggerMalware que registra cada tecla digitada pela vítima
L
LANLocal Area Network — rede dentro de uma área física limitada
least privilegeConceder apenas o acesso mínimo necessário — nada além disso
log filesRegistros do sistema de eventos — o que aconteceu e quando
loopEstrutura de código que repete um bloco de instruções (for, while)
M
MAC addressEndereço de hardware físico atribuído permanentemente a um dispositivo de rede
malwareQualquer software intencionalmente projetado para causar dano
memory dumpCaptura do conteúdo da RAM em um momento específico
MFAMulti-Factor Authentication — verificar identidade com 2 ou mais fatores
motherboardPlaca-mãe que conecta todos os componentes de hardware
mount pointLocal no sistema de arquivos onde um dispositivo de armazenamento é montado
N
NATNetwork Address Translation — mapeia IPs privados para um IP público
networkGrupo de dispositivos conectados que podem se comunicar entre si
NTFSFormato padrão de sistema de arquivos do Windows
O
open portUma porta ativamente escutando e aceitando conexões
OSOperating System — software que gerencia o hardware e todos os outros softwares
OSINTOpen Source Intelligence — coletar informações públicas sobre um alvo
OSI modelFramework de 7 camadas que explica como os dados trafegam por uma rede
OWASP Top 10Lista dos 10 riscos mais críticos de segurança em aplicações web
P
packetPequena unidade de dados que trafega independentemente por uma rede
pathEndereço completo de um arquivo ou pasta dentro do sistema de arquivos
path traversalUsar ../ na entrada para escapar dos diretórios permitidos e ler arquivos sensíveis
payloadA parte maliciosa de um ataque — o que realmente é executado
pentestSimulação autorizada de um ataque para encontrar vulnerabilidades
peripheralHardware externo conectado a um computador (impressora, mouse, USB)
permissionsRegras que definem quem pode ler, escrever ou executar um arquivo
persistenceManter acesso a um sistema mesmo após reinicialização
phishingEnviar e-mails falsos para enganar usuários a entregarem credenciais
PIDProcess ID — número único atribuído a cada processo em execução
portCanal numerado em que um serviço escuta conexões de entrada
post-exploitationAções tomadas após obter acesso — movimentação lateral, escalada de privilégios
pretextingCriar um cenário fabricado para extrair informações sensíveis
priv escalationIr de acesso limitado de usuário para controle total de admin ou root
processUm programa atualmente sendo executado pela CPU com seu próprio PID
process injectionEmbutir código malicioso dentro de um processo legítimo em execução
proxyIntermediário que fica entre o cliente e o servidor — pode interceptar tráfego
R
RAMRandom Access Memory — memória volátil rápida usada enquanto programas rodam
ransomwareMalware que criptografa arquivos da vítima e exige pagamento para descriptografar
reconReconnaissance — coletar informações sobre um alvo antes de atacar
red teamTime que simula atacantes reais em um exercício de segurança controlado
registersMemória minúscula e ultrarrápida dentro da CPU para dados atualmente ativos
rootConta de superusuário com acesso irrestrito ao sistema inteiro
rootkitMalware que se esconde fundo no OS para evitar detecção
routerDispositivo de rede que conecta redes diferentes e roteia tráfego
rwxRead, write, execute — os três tipos de permissão do Linux
S
scanningDescobrir portas abertas, serviços e vulnerabilidades em um alvo
sensitive data exposureDeixar dados confidenciais desprotegidos ou sem criptografia
serverO dispositivo ou programa que escuta e responde a requisições
sessionMecanismo do lado do servidor para rastrear um usuário autenticado
shellPrograma que interpreta e executa comandos do terminal
SIEMSecurity Information and Event Management — coleta e correlaciona logs
smishingAtaques de phishing conduzidos via mensagens SMS
SOCSecurity Operations Center — centro centralizado de defesa e monitoramento
social engineeringManipular pessoas psicologicamente para extrair informações ou acesso
softwareInstruções que dizem ao hardware o que fazer — código sem forma física
spear phishingPhishing altamente direcionado usando informações pessoais sobre a vítima
spywareMalware que monitora e coleta dados do usuário secretamente
SQL injectionInjetar código SQL em campos de entrada para manipular um banco de dados
SSRFServer-Side Request Forgery — fazer o servidor enviar requisições em seu nome
SSD / HDDDispositivos de armazenamento persistente — dados sobrevivem ao desligamento
SSL / TLSProtocolos criptográficos que protegem conexões HTTPS
stackParte da RAM usada para chamadas de função e armazenamento de variáveis locais
status codeCódigo de resposta numérico do servidor — 200 OK, 404 Não Encontrado, 500 Erro
subnetUma subdivisão de uma rede maior em segmentos lógicos menores
sudoComando Linux para executar um único comando com privilégios de root
switchDispositivo de rede que conecta múltiplos dispositivos na mesma rede
symmetric encryptionUsa a mesma chave para criptografar e descriptografar dados
T
tailgatingSeguir fisicamente alguém para uma área restrita sem autorização
TCPTransmission Control Protocol — garante entrega confiável e ordenada de pacotes
terminalInterface de texto para controlar o OS diretamente digitando comandos
threat huntingBuscar proativamente atacantes ocultos já dentro de uma rede
threat modelMapear o que pode dar errado, por quê, e contra o que se proteger
threadMenor unidade de execução dentro de um processo
trojanMalware disfarçado de software legítimo para enganar usuários a executá-lo
U
UDPUser Datagram Protocol — mais rápido que TCP mas sem garantia de entrega
URLUniform Resource Locator — endereço completo de um recurso web
userConta com acesso limitado no sistema — oposto de admin ou root
V
variableUm container nomeado que armazena um valor em um programa
virusMalware que se anexa a arquivos e se espalha quando executado
vishingAtaques de phishing conduzidos por chamadas de voz
volatile memoryMemória que é completamente apagada quando a energia é cortada (RAM)
VPNVirtual Private Network — túnel criptografado que protege o tráfego pela internet
vulnerabilityUma fraqueza em software, configuração ou hardware que pode ser explorada
W
WANWide Area Network — rede que abrange grandes distâncias geográficas
wormMalware autopropagate que não precisa de interação do usuário para se espalhar
X
XSSCross-Site Scripting — injetar scripts em páginas que outros usuários verão
Z
zero trustModelo de segurança — não confiar em nada por padrão, sempre verificar

IP Address
O endereço numérico único de um dispositivo em uma network.
W·02internetnetworking
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"O endereço de um dispositivo na rede — como um endereço residencial para computadores."
Cada dispositivo conectado a uma rede tem um endereço IP. Ele é usado para identificar quem está enviando dados e para onde enviar a resposta.
② Explicação

Existem duas versões. IPv4 usa 4 números separados por pontos: 192.168.0.1. Há cerca de 4 bilhões de endereços IPv4 possíveis, que estão se esgotando. IPv6 é a versão mais nova com um espaço de endereços muito maior: 2001:0db8:85a3::8a2e:0370:7334.

Private IPs (como 192.168.x.x) são usados dentro da sua rede doméstica ou do escritório. IPs públicos são os visíveis na internet. NAT traduz entre eles. Em segurança, seu endereço IP pode revelar sua localização aproximada e provedor — tornando rastreamento de IP e VPNs tópicos importantes.

③ Palavras-chave
IP addressIPv4IPv6private IPpublic IPNATsubnetlocalhost
④ Onde Aparece
Toda conexão de rede
Logs de servidor
Recon and scanning

⑤ Como Funciona
1Seu dispositivo solicita um IP ao roteador via DHCP
2Roteador atribui um IP privado (ex.: 192.168.1.5)
3Quando você vai online, seu provedor fornece um IP público
4NAT traduz entre seu IP privado e público
5Outros servidores veem seu IP público, não o privado
⑥ Ataque & Defesa
Ataque
  • Varredura de IP para encontrar alvos ativos
  • IP spoofing para falsificar o endereço de origem
  • Geolocalização via IP para perfilar alvos
Defesa
  • Use uma VPN para ocultar seu IP público
  • Regras de firewall para bloquear IPs suspeitos
  • Rate limiting e bloqueio de IP
⑦ Lembre-se
Seu IP público é visível para todo site que você visita. Ele pode revelar sua localização. Uma VPN mascara isso substituindo seu IP real pelo IP do servidor VPN.

DNS
O catálogo telefônico da internet — traduz nomes de domínio em endereços IP.
W·03internetprotocol
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"O sistema que traduz nomes de domínio em endereços IP."
Quando você digita google.com, seu computador não sabe o que isso significa — ele precisa de um IP address. DNS é o sistema que faz essa tradução automaticamente.
② Explicação

O DNS funciona em hierarquia. Primeiro seu computador verifica o cache local. Depois pergunta a um DNS resolver (geralmente seu provedor ou um público como o 8.8.8.8 do Google). O resolver sobe a cadeia — servidores raiz, servidores TLD (.com, .org), e por fim o nameserver autoritativo do domínio.

Registros DNS vêm em tipos: A record = domínio para IPv4, AAAA = domínio para IPv6, MX = servidor de server, CNAME = alias, TXT = verificação e outro texto. DNS é crítico em recon — consultar registros DNS revela detalhes da infraestrutura de um alvo.

③ Palavras-chave
DNSdomain nameDNS resolverA recordMX recordCNAMETXT recordTTLnameserver
⑤ Como Funciona
1Você digita google.com no navegador
2Computador verifica o cache DNS local
3Se não em cache, pergunta ao DNS resolver (ex.: 8.8.8.8)
4Resolver pergunta ao servidor raiz → servidor TLD → servidor autoritativo
5IP é retornado e o navegador se conecta a ele
⑥ Ataque & Defesa
Ataque
  • DNS spoofing — injetar respostas DNS falsas
  • Envenenamento de cache DNS — corromper o cache do resolver
  • Subdomain takeover — reivindicar subdomínios abandonados
  • DNS tunneling — esconder dados em consultas DNS
Defesa
  • Use DNSSEC para autenticar registros
  • Use resolvers confiáveis (1.1.1.1, 8.8.8.8)
  • Audite e remova subdomínios não usados
⑦ Lembre-se
DNS costuma ser a primeira coisa que um hacker sonda. Ferramentas como nslookup, dig e dnsx podem revelar muito sobre um alvo apenas pelos seus registros DNS.

HTTP / HTTPS
A linguagem que navegadores e servidores usam para trocar dados.
W·04internetprotocolweb
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"O protocolo que carrega toda página web que você já acessou."
HTTP (HyperText Transfer Protocol) é o conjunto de regras que navegadores e servidores seguem para solicitar e enviar conteúdo web. HTTPS é a mesma coisa mas com encryption — so no one in the middle can read what is being sent.
② Explicação

Quando você visita um site, seu navegador envia um HTTP request ao servidor. O servidor responde com um HTTP response contendo o conteúdo e um status code (200 = OK, 404 = não encontrado, 500 = erro do servidor).

HTTP é stateless — não lembra de você entre requisições. É por isso que cookies e sessions existem. HTTPS envolve o HTTP dentro de criptografia TLS — mesmo que alguém intercepte o tráfego, só vê dados incompreensíveis. O S no HTTPS significa Secure e em segurança essa diferença é tudo.

Métodos HTTP que você vai ver constantemente: GET (buscar dados), POST (enviar dados), PUT (atualizar), DELETE (remover), OPTIONS (verificar quais métodos são suportados).

③ Palavras-chave

1Navegador envia HTTP request: GET /page HTTP/1.1
2Request inclui headers (Host, User-Agent, cookies…)
3Servidor processa e envia de volta o HTTP response
4Response inclui status code + headers + body (a página)

⑥ Ataque & Defesa
Ataque
  • MITM em HTTP — interceptar tráfego não criptografado
  • SSL stripping — rebaixar HTTPS para HTTP
  • HTTP header injection
Defesa
  • Sempre use HTTPS — nunca envie dados sensíveis por HTTP
  • Ative HSTS (força HTTPS nos navegadores)
  • Use os flags secure e HttpOnly cookie flags
⑦ Lembre-se
Requests e responses HTTP são o batimento cardíaco da web. Uma vez que você os entende profundamente — métodos, headers, status codes — você começa a ver vulnerabilidades em todo lugar. Ferramentas como o Burp Suite vivem nessa camada.

URL
O endereço completo de qualquer recurso na web.
W·05internetweb
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"O endereço completo que diz ao navegador exatamente o que buscar e de onde."
URL significa Uniform Resource Locator. Toda página, imagem, arquivo e API endpoint na web tem uma URL única.
② Explicação

Uma URL tem várias partes: https://example.com:443/page?id=1#section

https = protocolo / example.com = domínio / :443 = porta (opcional, padrão para HTTPS) / /page = path / ?id=1 = query string / #section = fragment (âncora)

Em segurança, URLs são vetores de ataque. A query string (?id=1) é onde IDOR e path traversal começam. Atacantes também usam URLs para phishing — criando domínios falsos que parecem reais.

③ Palavras-chave
URLdomainpathquery stringparameterfragmentprotocolsubdomain
⑥ Ataque & Defesa
Ataque
  • Injetar payloads nos parâmetros de query
  • Path traversal pelo caminho da URL
  • Phishing com domínios parecidos com os reais
Defesa
  • Validar e sanitizar todos os parâmetros da URL
  • Usar allowlists para valores esperados
⑦ Lembre-se
O query string em uma URL é onde a maioria dos ataques web começa. Sempre trate parâmetros de URL como usuário não confiável.

Request & Response
Como o navegador e o servidor se comunicam.
W·06internetHTTP
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"O navegador pergunta (requisição). O servidor responde (resposta)."
Toda interação na web segue esse ciclo. O navegador envia um request com o que quer, e o servidor envia de volta um response com o conteúdo e um status code.
② Explicação

Um HTTP request tem: um método (GET, POST…), um path (/login), headers (metadados como cookies, User-Agent, Content-Type) e opcionalmente um body (dados sendo enviados).

Um HTTP response tem: status code (200, 404, 500…), headers (metadados como Content-Type, Set-Cookie) e um body (o HTML, JSON ou arquivo sendo retornado).

Em segurança, interceptar e modificar requests é a base dos testes web. Ferramentas como o Burp Suite ficam entre o navegador e o servidor e permitem ler, repetir e modificar cada request e response.

③ Palavras-chave
requestresponseHTTP methodheadersbodystatus codeBurp Suiteintercept
⑤ Como Funciona
1Navegador monta o request (método + path + headers + body)
2Request é enviado via TCP para o IP e port
3Server receives and processes the request
4Servidor envia de volta o response (status + headers + body)
5Browser renders the response content
⑥ Ataque & Defesa
Ataque
  • Interceptar e modificar requests para alterar parâmetros
  • Repetir requests para replicar ações
  • Fuzzing de requests para encontrar comportamentos inesperados
Defesa
  • Validar todos os dados do request no lado do servidor
  • Usar tokens CSRF para prevenir reenvio de requests
⑦ Lembre-se
Tudo que um app web faz é um request e um response. Se você consegue interceptá-los e modificá-los, consegue testar quase toda vulnerabilidade.

Status Codes
A resposta numérica do servidor dizendo o que aconteceu com seu request.
W·07internetHTTP
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Códigos de três dígitos que o servidor envia para dizer se sua requisição funcionou."
Todo response HTTP inclui um status code. São agrupados pelo primeiro dígito: 1xx = informativo, 2xx = sucesso, 3xx = redirecionamento, 4xx = erro do cliente, 5xx = erro do servidor.
② Explicação

Os mais importantes para saber: 200 OK = sucesso. 201 Created = recurso criado. 301/302 = redirecionamento (permanente/temporário). 400 Bad Request = request malformado. 401 Unauthorized = não autenticado. 403 Forbidden = autenticado mas sem permissão. 404 Not Found = recurso não existe. 500 Internal Server Error = algo quebrou no servidor.

Em segurança, status codes revelam muita coisa. Um 403 significa que o recurso existe mas está bloqueado. Um 200broken auth. Um 500 pode indicar que uma injeção funcionou.

③ Palavras-chave
status code200 OK301 redirect401 unauthorized403 forbidden404 not found500 server error
⑥ Ataque & Defesa
Ataque
  • 403 revela que o recurso existe — tente contornar
  • 500 em input de injeção = possível vulnerability
  • Códigos diferentes para usuários válidos vs inválidos = enumeração
Defesa
  • Retornar mensagens de erro genéricas
  • Não revelar detalhes internos em erros 500
⑦ Lembre-se
Status codes são pistas. Um hacker os lê como um mapa — cada código diz algo sobre o que está no servidor e como ele se comporta.

Headers
Metadados enviados junto a todo request e response HTTP.
W·08internetHTTP
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Informação extra anexada a cada requisição e resposta HTTP."
Headers são pares chave-valor que carregam metadados — coisas como qual tipo de conteúdo está sendo enviado, quem está enviando, quais cookies existem e como o response deve ser armazenado em cache.
② Explicação

Headers de request comuns: Host (qual domínio você quer), User-Agent (qual navegador você está usando), Cookiesession), Content-Type (formato do body), Authorization (seu token de autenticação).

Headers de response comuns: Content-Type (formato do response), Set-Cookie (servidor definindo um cookie), Location (destino do redirecionamento), X-Frame-Options (previne clickjacking), Content-Security-Policy (regras de proteção contra XSS).

Em segurança, headers vazam informações e podem ser manipulados. Headers de segurança faltando são um achado comum em pentests e bug bounties.

③ Palavras-chave
headersHostUser-AgentCookieAuthorizationContent-TypeSet-CookieCSPHSTS
⑥ Ataque & Defesa
Ataque
  • Injeção de Host header para manipular roteamento
  • Headers de segurança faltando habilitam XSS ou clickjacking
  • Vazar informações do servidor via headers Server e X-Powered-By
Defesa
  • Adicionar headers de segurança: CSP, HSTS, X-Frame-Options
  • Remover headers que vazam informações do servidor
  • Validar o Host header no lado do servidor
⑦ Lembre-se
Headers de segurança são proteções gratuitas. CSP, HSTS ou X-Frame-Options faltando é um achado em quase todo pentest. Sempre verifique quais headers um servidor retorna.

Cookies
Pequenos arquivos armazenados no navegador para lembrar o estado do usuário entre requests.
W·09internetsessions
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Pequenos dados que o servidor armazena no seu navegador para lembrar de você."
HTTP é stateless — ele esquece você após cada request. Cookies resolvem isso armazenando um pequeno valor no navegador que é enviado com cada request futuro para aquele site.
② Explicação

Quando você faz login, o servidor cria uma session e envia de volta um cookie contendo seu session ID. Seu navegador armazena e envia esse cookie com cada request subsequente — é assim que o servidor sabe quem você é.

Atributos de cookie importam em segurança: HttpOnly impede que JavaScript leia o cookie (bloqueia roubo de cookie via XSS). Secure garante que o cookie só é enviado via HTTPS. SameSite protege contra CSRF. Flags faltando são vulnerabilidades muito comuns.

③ Palavras-chave
cookiesession IDHttpOnlySecure flagSameSiteSet-Cookiecookie theftsession hijacking
⑥ Ataque & Defesa
Ataque
  • XSS para roubar o session cookie = tomada de conta
  • Session fixation — forçar a vítima a usar um session ID conhecido
  • Session hijacking via cookie interceptado
Defesa
  • Definir flag HttpOnly — JavaScript não consegue ler o cookie
  • Definir flag Secure — enviado apenas via HTTPS
  • Definir SameSite=Strict para prevenir CSRF
⑦ Lembre-se
Roubar um session cookie = roubar a sessão logada de alguém. É por isso que XSS é tão perigoso — ele pode extrair cookies silenciosamente e entregá-los ao atacante.

① Definição Simples

Vulnerabilidades de gerenciamento de session estão no OWASP Top 10. Problemas comuns: session IDs que não expiram, session IDs previsíveis, sessions não invalidadas após logout e ataques de session fixation.

Defesa
  • Regenerar o session ID após o login
  • Definir expiração de session e timeouts de inatividade
  • Invalidar a session completamente no logout
⑦ Lembre-se
O session ID é a joia da coroa. Proteja-o como uma senha — porque é efetivamente a mesma coisa.

Client & Server
Quem pergunta, quem responde — o modelo fundamental da web.
W·12internetarchitecture
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Client = quem pergunta. Server = quem responde."
Em toda interação web, um client faz um request e um server responde. Seu navegador é o client. O computador que hospeda o site é o server.
② Explicação

O client inicia a comunicação — envia requests e renderiza os responses. O server escuta requests recebidos e envia respostas adequadas. Um único servidor pode atender milhares de clients simultaneamente.

Em segurança, essa distinção importa porque: código client-side (HTML, JavaScript, CSS) pode sempre ser visualizado e modificado pelo usuário — nunca confie nele. Código server-side roda no servidor e não é visível aos usuários — é onde a validação real deve acontecer. Nunca confie em validação de input client-side.

③ Palavras-chave
clientserverclient-sideserver-sidebrowserweb serverrequestresponse
⑥ Ataque & Defesa
Ataque
  • Bypassar validação client-side — é trivialmente removível
  • Manipular JavaScript para remover restrições
  • Enviar requests que a UI não foi projetada para permitir
Defesa
  • Sempre validar input no lado do servidor
  • Nunca depender de verificações client-side para segurança
⑦ Lembre-se
Qualquer coisa que o client envia pode ser falsificada. Segurança real só existe no servidor. Validação client-side é para experiência do usuário, não para segurança.

  • Usar regras de WAF do CDN para filtrar tráfego malicioso
⑦ Lembre-se

SSL / TLS
A camada de criptografia que torna o HTTPS seguro.
W·14internetencryptionHTTPS
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"A tecnologia que criptografa dados viajando entre seu navegador e um servidor."
SSL (agora substituído pelo TLS) é o protocolo criptográfico que faz o HTTPS funcionar. Garante que a conexão é criptografada e que o servidor é quem afirma ser.
② Explicação

SSL (Secure Sockets Layer) é o nome antigo — está descontinuado e é inseguro. TLS (Transport Layer Security) é o padrão atual. Quando as pessoas falam SSL geralmente querem dizer TLS.

TLS oferece três coisas: criptografia (seus dados não podem ser lidos em trânsito), autenticação (o servidor prova sua identidade via certificado HTTPS) e integridade (dados não podem ser adulterados em trânsito). O handshake TLS acontece antes de qualquer dado HTTP ser trocado — ele estabelece o túnel criptografado primeiro.

Em segurança, versões antigas de TLS (SSL 2/3, TLS 1.0/1.1) são vulneráveis. Ferramentas como Nmap e testssl.sh verificam quais versões um servidor suporta.

③ Palavras-chave
SSLTLSHTTPScertificatehandshakeencryptionCAcipher suitePKI
⑤ Como Funciona
1Client conecta ao servidor e envia as versões TLS suportadas
2Servidor responde com seu ertificado e o cipher escolhido
3Client verifica o certificado (é válido? emitido por CA confiável?)
4Trocam chaves de criptografia usando criptografia assimétrica
5Túnel criptografado estabelecido — dados HTTP fluem dentro dele
⑥ Ataque & Defesa
Ataque
  • SSL stripping — rebaixar HTTPS para HTTP
  • Cipher suites fracos — quebrar criptografia mais antiga
  • Certificados inválidos ou autoassinados podem ser falsificados
Defesa
  • Usar TLS 1.2 ou 1.3 no mínimo
  • Ativar HSTS para prevenir SSL stripping
  • Usar certificados de Certificate Authorities confiáveis
⑦ Lembre-se
HTTPS sem TLS válido não é seguro. O cadeado no navegador apenas significa que a conexão é criptografada — não significa que o site é seguro ou legítimo.

Bloco 1 — resumos visuais rápidos de cada termo em ordem.
W·01 — Internet vs Web
Internet = a rede global de computadores. Web = um serviço que roda em cima dela usando HTTP.
Internet = estradas. Web = um tipo de carro. Email e SSH também usam as estradas.
Não é a mesma coisa. A web é só uma parte da internet.
W·02 — IP Address
O endereço único de um dispositivo na rede. IPv4 = 4 números. IPv6 = mais longo.
Dispositivo liga → DHCP atribui IP privado → NAT traduz pra IP público → internet enxerga o IP público
Seu IP público é visível pra todo site que você visita. VPN mascara isso.
W·03 — DNS
Traduz nomes de domínio em IPs. A agenda telefônica da internet.
Você digita google.com → cache local → DNS resolver → root → TLD → authoritative → retorna o IP
DNS é o primeiro lugar que um hacker investiga. Revela muito sobre a infraestrutura.
W·04 — HTTP / HTTPS
O protocolo que carrega páginas web. HTTP = não criptografado. HTTPS = criptografado com TLS.
Browser envia request → servidor processa → envia response com status code e conteúdo
HTTP é a camada onde quase todos os ataques web acontecem. Aprenda bem.
W·05 — URL
O endereço completo de um recurso web. Protocolo + domínio + caminho + query string.
https://site.com/page?id=1 → protocol / domain / path / query string
O query string é onde a maioria dos ataques começa. Sempre trate parâmetros como input não confiável.
W·06 — Request and Response
Browser pergunta (request). Servidor responde (response). Cada interação na web segue esse ciclo.
Request: method + path + headers + body → Response: status code + headers + body
Se você consegue interceptar e modificar requests, consegue testar quase toda vulnerabilidade web.
W·07 — Status Codes
O código numérico que o servidor retorna dizendo o que aconteceu com o request.
2xx = request succeeded · 3xx = redirect · 4xx = erro do cliente · 5xx = erro do servidor
Status codes são pistas. 403 = recurso existe mas bloqueado. 500 em injection = possível vuln.
W·08 — Headers
Metadados enviados em todo request e response. Carregam cookies, tipos de conteúdo, regras de segurança.
Request headers: Host, User-Agent, Cookie · Response headers: Set-Cookie, CSP, HSTS
Headers de segurança faltando são um achado em quase todo pentest. Sempre cheque.
W·09 — Cookies
Pequenos dados que o servidor armazena no browser para lembrar o usuário entre requests.
Login → servidor cria sessão → envia cookie com session ID → browser manda em todo request futuro
Roubar o session cookie = roubar a sessão logada. É por isso que XSS é tão perigoso.
W·10 — Sessions
Registro server-side que mantém o estado do usuário logado. O session ID no cookie é a chave.
Login → session criada no servidor → ID enviado via cookie → cada request verifica o ID
O session ID é a coroa. Proteja como uma senha — porque é exatamente isso.
W·11 — API
Interface que deixa programas se comunicarem. REST APIs usam HTTP e retornam JSON.
App envia GET /weather → API retorna JSON com dados → app exibe pro usuário
APIs são a nova superfície de ataque. Sempre teste a API quando testar uma aplicação web.
W·12 — Client and Server
Cliente = quem pede. Servidor = quem responde. Código client-side pode sempre ser modificado.
Browser (client) → envia request → servidor processa → envia response → browser renderiza
Nunca confie em validação client-side. Segurança real só existe no servidor.
W·13 — CDN
Rede de servidores distribuídos que entrega conteúdo do ponto mais próximo do usuário.
Usuário requisita → CDN verifica cache → serve do edge server mais próximo ou busca do origin
Encontrar o IP do servidor de origem bypassa o CDN e a WAF que o protege.
W·14 — SSL / TLS
O protocolo criptográfico que faz o HTTPS funcionar. Garante encriptação, autenticação e integridade.
Client conecta → handshake → troca de chaves → túnel criptografado estabelecido → HTTP flui dentro
O cadeado no browser só significa que a conexão é criptografada — não que o site é seguro.

Network
O conceito fundamental sobre o qual tudo mais se constrói.
N·01networkfundamentals
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Dois ou mais dispositivos que podem se comunicar formam uma rede."
Uma rede é qualquer grupo de dispositivos conectados que compartilham dados. Dois laptops no mesmo WiFi formam uma rede. A internet é uma rede de redes.
② Explicação

Todo dispositivo numa rede tem uma identidade (um IP address) e usa regras acordadas (protocolos) para se comunicar. Sem uma rede, computadores são ilhas isoladas. Com uma, eles podem compartilhar arquivos, serviços e ataques.

Redes existem em tamanhos diferentes: um único WiFi doméstico (LAN), uma rede corporativa abrangendo prédios (WAN), ou a internet inteira. Os princípios de como funcionam são os mesmos — apenas a escala muda.

③ Palavras-chave
networkdeviceconnectioncommunicationLANWANtopology
④ Onde Aparece
Todo dispositivo conectado — celular, laptop, IoT
WiFi de escritório e doméstico
Serviços em nuvem e data centers
Todo ciberataque começa em uma rede
⑤ Como Funciona
Dispositivos se conectam via links cabeados (Ethernet) ou sem fio (WiFi, celular). Cada dispositivo recebe um IP address. Quando dados precisam se mover, são quebrados em pacotes, cada um rotulado com informações de destino. Roteadores e switches encaminham pacotes pelo path. O dispositivo receptor os remonta.
⑥ Ataque & Defesa
Ataque
  • Sniffing — capturar tráfego não criptografado na mesma rede
  • Movimento lateral — usar um dispositivo comprometido para alcançar outros
  • Mapeamento de rede com ferramentas como Nmap para encontrar hosts ativos
Defesa
  • Criptografar todo tráfego — TLS, SSH, para anomalias
⑦ Lembre-se
Se você consegue alcançar um dispositivo pela rede, pode atacá-lo. Se não consegue alcançá-lo, não pode. Acesso à rede é o pré-requisito para tudo mais.

LAN vs WAN
Local versus ampla — dois modelos de segurança diferentes.
N·02LANWAN
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"LAN = sua casa. WAN = a cidade. Internet = o mundo."
Uma LAN (Local Area Network) cobre uma área limitada como uma casa, escritório ou prédio. Uma WAN (Wide Area Network) conecta múltiplas LANs através de cidades, países ou o planeta inteiro — a própria internet é a maior WAN.
② Explicação

Dentro de uma LAN, dispositivos geralmente confiam uns nos outros e o tráfego flui livremente. A fronteira entre LAN e WAN é onde firewalls e roteadores ficam, decidindo o que entra e sai.

Essa diferença de confiança importa. Muitos serviços internos têm autenticação fraca porque assumem "você já está dentro, então é confiável". É por isso que entrar numa LAN corporativa — mesmo que brevemente — é tão valioso para um atacante.

③ Palavras-chave
LANWANsubnetrouterfirewallprivate networkpublic network
④ Onde Aparece
WiFi doméstico = LAN
Rede de escritório = LAN conectada à WAN
Internet = a WAN global
Redes celulares conectam dispositivos à WAN
⑤ Como Funciona
Dispositivos numa LAN geralmente compartilham um range de IP privado (192.168.x.x ou 10.x.x.x) e se comunicam diretamente via switches. Para alcançar a WAN, o tráfego passa por um roteador que traduz IPs privados para um público (NAT). Firewalls na fronteira LAN/WAN filtram o que é permitido.
⑥ Ataque & Defesa
Ataque
  • Uma vez dentro da LAN, o atacante tem visibilidade de muitos serviços internos
  • Serviços expostos na WAN são os primeiros alvos de qualquer atacante externo
  • Pivoting — usar um dispositivo LAN comprometido para atacar outros dispositivos LAN
Defesa
  • Zero trust internamente — nunca assuma que tráfego LAN é seguro
  • Firewalls fortes entre LAN e WAN
  • Microsegmentação — dividir a LAN em zonas de confiança menores
⑦ Lembre-se
A fronteira LAN/WAN é crítica, mas assumir que tudo dentro da LAN é seguro é o erro mais comum. Trate o tráfego interno com ceticismo também.

"Sete camadas empilhadas: Physical, Data Link, Network, Transport, Session, Presentation, Application."

Ataques diferentes visam camadas diferentes — saber em qual camada algo acontece diz quais defesas se aplicam.

TCP
Entrega confiável e ordenada — o pilar da internet.
N·04TCPtransport
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"TCP é o protocolo que garante que seus dados cheguem, em ordem, com erros detectados."
TCP (Transmission Control Protocol) é um protocolo baseado em conexão que garante que os dados chegam de forma confiável e na ordem correta. Usado por navegação web, e-mail, SSH e a maioria das coisas onde dados perdidos seriam um problema.
② Explicação

TCP começa com um three-way handshake: SYN (client diz olá), SYN-ACK (server responde), ACK (client confirma). Após isso, ambos os lados sabem que a conexão está estabelecida e podem trocar dados com confiança.

TCP numera cada byte enviado. Se algo se perde, é retransmitido. Se pacotes chegam fora de ordem, o TCP os reordena. Essa confiabilidade tem um custo — o TCP é mais lento que o UDP por causa do controle. Mas para coisas que importam (como carregar uma página), confiabilidade supera velocidade.

③ Palavras-chave
TCPthree-way handshakeSYNSYN-ACKACKreliableconnection-oriented
④ Onde Aparece
HTTP/HTTPS websites
Acesso remoto via SSH
E-mail SMTP/IMAP
Transferência de arquivos FTP
⑤ Como Funciona
Setup da conexão usa o handshake SYN/SYN-ACK/ACK. Uma vez estabelecida, dados são quebrados em segmentos numerados sequencialmente. Cada segmento requer confirmação. Segmentos perdidos são retransmitidos. A conexão termina com pacotes FIN de ambos os lados.
⑥ Ataque & Defesa
Ataque
  • SYN flood — enviar muitos SYN sem completar o handshake, esgotando slots de conexão
  • TCP RST injection — forjar pacotes de reset para derrubar conexões legítimas
  • Previsão de número de sequência (majoritariamente histórico, mas atenção em implementações fracas)
Defesa
  • SYN cookies — lidar com SYN floods sem manter estado
  • Rate limiting de conexões por IP de origem
  • Randomização forte de números de sequência
⑦ Lembre-se
TCP significa confiabilidade. Use quando dados perdidos quebrariam sua aplicação — sites, transferências de arquivo, logins.

UDP
Rápido, leve, sem garantias.
N·05UDPtransport
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"UDP dispara e esquece — rápido mas não confiável."
UDP (NS lookups, VoIP.quot;)">User Datagram Protocol) é sem conexão e sem estado. Simplesmente envia pacotes sem confirmação. Usado quando velocidade importa mais que confiabilidade — games, streaming de vídeo, consultas NS lookups, VoIP.quot;)">DNS, VoIP.
② Explicação

UDP não tem handshake, retransmissão ou ordenação. Você envia um acote e torce para que chegue. A troca é muito menos overhead — perfeito para coisas onde dados tardios são piores que dados perdidos.

Por exemplo, em uma videochamada ao vivo, retransmitir um frame perdido só criaria lag. Melhor pulá-lo e continuar. O mesmo com games online — a posição antiga de um segundo atrás não importa, a atual sim. A simplicidade do UDP também o torna atrativo para ataques de amplificação.

③ Palavras-chave
UDPconnectionlessstatelessdatagramfastunreliable
④ Onde Aparece
Consultas DNS (orta 53)
Chamadas VoIP
Games online
Streaming de vídeo
Protocolos de descoberta de rede
⑤ Como Funciona
Sem setup de conexão. Cada pacote (datagrama) é independente. O remetente envia, o receptor recebe — ou não. Sem confirmações, sem retransmissão. A camada de aplicação lida com qualquer confiabilidade necessária (ou aceita a perda).
⑥ Ataque & Defesa
Ataque
  • UDP amplificação — request pequeno, response enorme (DNS, NTP, memcached) direcionado à vítima
  • Falsificar IP de origem é trivial pois não há handshake para verificar
  • UDP flood — sobrecarregar o alvo com pacotes lixo
Defesa
  • Rate limiting em serviços baseados em UDP
  • Filtros anti-spoofing na borda da rede (BCP 38)
  • Desabilitar resolvers DNS abertos e vetores de amplificação
⑦ Lembre-se
UDP significa rápido e arriscado. Ótimo para tempo real, terrível se confiabilidade importa. A maioria dos ataques de amplificação usa UDP porque não pode ser verificado.

MAC Address
O endereço físico gravado em todo cartão de rede.
N·06MAClayer 2
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Sua placa de rede tem um MAC único. Roteadores veem IPs, switches veem MACs."
Um endereço MAC é um identificador de 48 bits atribuído a uma placa de interface de rede (NIC). Formato: 6 pares hex como AA:BB:CC:DD:EE:FF. É o endereço físico usado na Camada 2 — dentro de uma rede local, dispositivos se encontram por MAC.
② Explicação

Endereços MAC são teoricamente únicos e atribuídos pelos fabricantes (os primeiros 3 octetos identificam o vendedor). Porém, quase todo OS moderno permite mudar seu MAC — útil para privacidade em WiFi público, e também para atacantes. Endereços MAC só importam dentro de um segmento de rede local. Quando o tráfego cruza um roteador, MACs são removidos e substituídos. Então um MAC diz quem está na rede local — nunca nada sobre atacantes distantes.

③ Palavras-chave
MAC addressNICOUIlayer 2physical addressARP
④ Onde Aparece
Ferramentas de admin de rede (ifconfig, ip link, ipconfig)
Logs de acesso WiFi
DHCP servidor de registros
Sistemas de controle de acesso à rede (NAC)
⑤ Como Funciona
Cada pacote de rede em uma LAN tem MAC de origem, MAC de destino, IP de origem, IP de destino. Switches encaminham com base em tabelas MAC que constroem observando o tráfego. ARP (Address Resolution Protocol) mapeia IPs para MACs para que a rede saiba onde entregar.
⑥ Ataque & Defesa
Ataque
  • MAC spoofing — mudar seu MAC para bypassar filtragem baseada em MAC ou se passar por outro dispositivo
  • ARP spoofing — alegar ser o gateway, interceptando todo tráfego
  • MAC flooding — encher a tabela CAM do switch para forçar modo hub (transmite tudo)
Defesa
  • Port security em switches — limitar MACs por porta
  • Dynamic ARP Inspection (DAI) — bloquear respostas ARP inválidas
  • Autenticação 802.1X — verificar identidade antes de conceder acesso à rede
⑦ Lembre-se
MAC = identidade local. Fácil de falsificar, mas só importa no mesmo segmento de rede. A maioria dos ataques de switching visam MACs.

Router
O tomador de decisões que conecta redes.
N·07routerlayer 3
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Um roteador decide para qual rede um pacote deve ir a seguir."
Um roteador conecta múltiplas redes e decide para onde enviar cada pacote. Operando na Camada 3, lê endereços IP de destino e encaminha o tráfego usando uma tabela de roteamento.
② Explicação

Seu roteador doméstico combina várias funções: roteamento, firewall, servidor DHCP, resolver DNS e ponto de acesso WiFi. Redes corporativas geralmente separam essas funções em dispositivos diferentes.

A internet é essencialmente uma enorme rede de roteadores passando pacotes entre si. Cada roteador conhece seus vizinhos e usa protocolos como BGP para aprender caminhos. Se o roteamento falha ou é manipulado, o tráfego pode ser desviado — um grande vetor de ataque contra infraestruturas grandes.

③ Palavras-chave
routergatewayrouting tableNATBGPOSPFdefault gateway
④ Onde Aparece
Roteadores WiFi domésticos e de escritório
Equipamento backbone de ISP
VPCs e redes virtuais em nuvem
O núcleo da internet
⑤ Como Funciona
Cada interface de rede num roteador tem seu próprio IP. Quando um pacote chega, o roteador verifica o IP de destino na tabela de roteamento para encontrar o próximo hop. Reescreve os endereços MAC de origem/destino (pois o próximo hop é um segmento L2 diferente), decrementa o TTL e encaminha. Pacotes sem rota correspondente vão para o gateway padrão — geralmente o ISP upstream.
⑥ Ataque & Defesa
Ataque
  • Credenciais padrão em roteadores domésticos — admin/admin ainda funciona em milhões
  • N
  • Para empresas — RPKI para validar rotas BGP
⑦ Lembre-se
Roteador comprometido = rede inteiramente comprometida. Quem controla o roteamento controla quem vê o quê. Credenciais padrão ainda são a fraqueza #1 dos roteadores.

Switch
O diretor de tráfego dentro de uma rede.
N·08switchlayer 2
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Um switch encaminha dados dentro de uma única rede — por MAC, não IP."
Um switch conecta dispositivos dentro do mesmo segmento de rede, encaminhando frames com base em endereços MAC. Diferente de um hub (que transmite tudo), um switch aprende quais dispositivos estão em qual orta e envia dados apenas para onde precisam ir.
② Explicação

Switches são os heróis anônimos das redes de escritório e data center. Fazem as redes cabeadas funcionar roteando frames inteligentemente — apenas o destinatário pretendido vê os dados, não todo dispositivo conectado.

Switches modernos também suportam VLANs (separação lógica de segmentos de rede), Quality of Service e controle de acesso. Comprometer ou enganar um switch dá a um atacante visibilidade sobre tráfego que normalmente seria invisível de outra parte da rede.

③ Palavras-chave

Firewall
O segurança do tráfego de rede.
N·09firewallsecurity
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Um firewall aplica regras sobre qual tráfego pode passar."
Um firewall é um dispositivo de segurança ou oftware que filtra tráfego de rede com base em regras. Algum tráfego é permitido, algum bloqueado, algum registrado. É a primeira linha de defesa entre redes confiáveis e não confiáveis.
② Explicação

Firewalls vão de simples filtros de pacotes (esse pacote corresponde a um IP e porta de origem permitidos?) a firewalls de próxima geração avançados (NGFW) que inspecionam conteúdo de aplicação, descriptografam TLS e bloqueiam padrões de malware conhecidos.

Um firewall é tão bom quanto suas regras. Um firewall mal configurado com regras excessivamente permissivas fornece falsa segurança. A melhor prática é "deny by default" — bloquear tudo por padrão e só permitir o que é especificamente necessário.

③ Palavras-chave

④ Onde Aparece
Fronteira entre N
Grupos de segurança de nuvem (AWS, Azure)

⑤ Como Funciona
Quando um pacote chega ao firewall, é verificado contra o conjunto de regras. Regras geralmente incluem IP de origem/destino, porta, protocolo. Firewalls stateful lembram estados de conexão — permitindo tráfego de retorno para conexões estabelecidas sem regras explícitas. NGFWs vão mais fundo, inspecionando conteúdo da camada de aplicação.
⑥ Ataque & Defesa
Ataque
  • Bypass via portas permitidas — tunelizar tráfego proibido via TTPS (443) que quase sempre está aberta
  • Regras mal configuradas — regras de permissão excessivamente amplas que anulam o propósito do firewall
  • Dentro do firewall — uma vez comprometido, o movimento lateral não enfrenta firewall
Defesa
  • Default deny — permitir explicitamente apenas tráfego necessário
  • Auditorias regulares de regras — remover regras obsoletas
  • Filtragem de saída — limitar o que pode sair, não apenas o que pode entrar
  • Firewalls em camadas — regras diferentes no perímetro, segmentos internos, host
⑦ Lembre-se
Firewall = regras. Regras ruins = sem firewall. O padrão "permitir tudo de dentro" derrota completamente o propósito.

Ports & Protocols
O sistema de endereçamento dentro de um IP address.

tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Porta = um serviço específico em um host. O IP te leva ao prédio, a porta te leva ao escritório certo."
Uma porta é um endpoint de comunicação numerado em um dispositivo, variando de 0 a 65535. Combinada com um endereço IP, identifica um serviço específico. Portas comuns incluem SSH (22), HTTP (80), HTTPS (443), DNS (53). Porta + protocolo (TCP/UDP) define qual serviço roda lá.
② Explicação

Portas abaixo de 1024 são "well-known" e requerem privilégios de admin para vincular na maioria dos sistemas. Estas são reservadas para serviços padrão. Portas mais altas são "efêmeras" — portas temporárias usadas por clients para conexões de saída.

Para atacantes, o primeiro passo é quase sempre o escaneamento de portas — encontrar quais portas estão abertas num alvo. Cada porta aberta é uma potencial superfície de ataque. Para defensores, o princípio é o oposto: fechar toda porta que não precisa estar aberta.

⑤ Como Funciona
Quando um serviço inicia, ele "vincula" a uma porta e começa a escutar. Quando o tráfego chega naquele IP e porta, o S o roteia para a aplicação. Múltiplos serviços podem rodar em portas diferentes do mesmo host. Portas fechadas respondem com RST (TCP) ou ICMP unreachable. Portas filtradas não respondem.
⑥ Ataque & Defesa
Ataque
  • Escaneamento de portas para encontrar serviços abertos (nmap, masscan)
  • Banner grabbing — conectar a portas para identificar versões de oftware
  • Explorar vulnerabilidades conhecidas em serviços rodando em portas abertas
Defesa
  • Fechar toda porta desnecessária — minimizar superfície de ataque
  • Usar portas não padrão para SSH e serviços admin (segurança por obscuridade é fraca sozinha, mas útil como camada adicional)
  • Monitorar para port scans — alertar em padrões de scan incomuns
⑦ Lembre-se
Porta aberta = entrada. Toda porta aberta deve justificar sua existência. A primeira coisa que todo atacante faz é enumerar portas.

Packet
O átomo da comunicação de rede.
N·11packetfundamentals
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Todo byte que você envia vira um pacote — um pequeno pacote com informações de destino."
Um pacote é uma pequena unidade de dados que viaja independentemente por uma rede. Contém um payload (os dados reais) e headers (metadados como IP de origem, IP de destino, porta, protocolo). Arquivos grandes viram muitos pacotes que são remontados no destino.
② Explicação

Quando você envia um arquivo de 100MB, a rede não envia 100MB de uma vez. É quebrado em muitos pacotes — cada um tipicamente 1500 bytes (o MTU padrão). Cada pacote encontra seu próprio caminho pela rede e podem até chegar fora de ordem. O destino os remonta.

Esse design baseado em pacotes é o que torna a internet resiliente. Se um caminho falha, os pacotes encontram outro. Também é o que torna as redes "sniffáveis" — pacotes viajam em texto claro (a menos que criptografados) e qualquer um com acesso ao caminho pode lê-los.

③ Palavras-chave
packetheaderpayloadfragmentMTUframedatagram
④ Onde Aparece
Wireshark e capturas de pacotes
Troubleshooting de rede
Análise forense
Detecção de padrões de tráfego maliciosos
⑤ Como Funciona
Um pacote é construído camada por camada (encapsulação): os dados da aplicação são envolvidos com um header TCP/UDP, depois um header IP, depois um frame header Layer 2 (Ethernet/MAC). Cada hop desencapsula e re-encapsula conforme necessário.
⑥ Ataque & Defesa
Ataque
  • Sniffing — capturar pacotes que atravessam a rede (especialmente em mídia compartilhada)
  • Injeção de pacotes — injetar pacotes maliciosos para manipular conexões
  • Ataques de replay — capturar pacotes legítimos e reenviá-los depois
Defesa
  • Criptografar todo tráfego — TLS, SSH, — detectar padrões de pacotes anômalos
⑦ Lembre-se
Pacotes são fundamentais — todo ataque e defesa opera em pacotes no final. Wireshark mostra exatamente o que está no cabo.

Bandwidth
Quanta informação pode fluir por segundo.
N·12bandwidthcapacity
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Bandwidth é o tamanho do cano. Mais bandwidth = mais dados simultâneos."
Bandwidth é a quantidade máxima de dados que pode transferir por uma conexão de ede num tempo determinado, geralmente medida em bits por segundo (Mbps, Gbps). É a capacidade da rede — não quão rápido pacotes individuais viajam, mas quantos podem fluir ao mesmo tempo.
② Explicação

Termos confusos: bandwidth é capacidade, throughput é taxa de transferência real alcançada, latência é o atraso entre request e response. Uma conexão pode ter alta bandwidth mas alta latência (internet via satélite) ou baixa bandwidth mas baixa latência (DSL antigo).

Em segurança, bandwidth importa porque esgotá-la é o objetivo completo dos ataques DDoS volumétricos. Se um alvo tem 1 Gbps de bandwidth e um atacante pode inundar com 10 Gbps de lixo, tráfego legítimo não consegue passar. O defensor não precisa bloquear o ataque — apenas precisa absorvê-lo.

③ Palavras-chave
bandwidththroughputMbpsGbpscapacitysaturationDDoS
④ Onde Aparece
Medições de velocidade de internet
Cálculos de custo em nuvem (egress é cobrado)
Design de rede e planejamento de capacidade
Métricas de ataques DDoS
⑤ Como Funciona
Redes têm máximos físicos ditados pelo meio (cobre, fibra, sem fio) e pelo equipamento. Além disso, gerenciamento de congestionamento entra — pacotes são enfileirados ou descartados. Bandwidth é compartilhada entre usuários num link, então 100 Mbps compartilhados por 100 usuários dá 1 Mbps cada no pior caso.
⑥ Ataque & Defesa
Ataque
  • DDoS volumétrico — inundar o alvo com mais tráfego do que sua bandwidth consegue absorver
  • Ataques de amplificação — usar protocolos como NS ou NTP para multiplicar o volume do ataque
  • Slowloris — oposto do volumétrico, esgotar conexões sem alta bandwidth
Defesa
  • Serviços de proteção DDoS (Cloudflare, AWS Shield) absorvem ataques em escala
  • Redes anycast distribuem tráfego por muitos pontos de presença
  • Rate limiting e traffic shaping no perímetro
⑦ Lembre-se
Bandwidth é o gargalo. A maioria dos incidentes "o site está fora" são saturação de bandwidth, seja de ataque ou tráfego legítimo inesperado.

Proxy

N·13proxymiddleware
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Um proxy fala em nome de outra pessoa — ele encaminha requisições em seu nome."
Um proxy é um servidor que fica entre um client e outro servidor, encaminhando requests e responses. Pode esconder a identidade do client, fazer cache de conteúdo, filtrar conteúdo ou inspecionar tráfego. Forward proxies agem em nome de clients; reverse proxies agem em nome de servidores.

Forward proxies (como proxies de filtragem corporativos) escondem de onde os requests vêm, aplicando políticas no tráfego de saída. Reverse proxies (como Cloudflare, nginx) escondem onde os servidores estão, distribuindo carga e adicionando segurança.

Em testes de segurança, um proxy de interceptação como Burp Suite é essencial. Fica entre seu navegador e o alvo, permitindo ver, modificar e repetir cada request. É assim que a maioria dos testes de segurança web acontece — inspecionando e manipulando requests em trânsito.

NAT
Network Address Translation — fazendo muitos parecerem um só.
N·15NATaddress translation
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"NAT permite que vários dispositivos compartilhem um IP público — e por acidente oferece alguma proteção."
NAT (Network Address Translation) é a técnica de mapear múltiplos endereços IP privados para um único IP address público. Quase todo roteador doméstico faz isso — seu celular, notebook e TV compartilham um IP público do seu provedor. O NAT foi inventado porque o IPv4 ficou sem endereços.
② Explicação

NAT funciona rastreando conexões de saída. Quando seu laptop (192.168.1.5) envia dados para o google.com, o roteador reescreve o IP de origem para o IP público e armazena o mapeamento. Quando o Google responde, o roteador usa esse mapeamento para encaminhar a resposta de volta ao seu laptop.

Um efeito colateral do NAT: conexões de entrada vindas da internet não têm para onde ir por padrão — elas não saberiam qual dispositivo interno acessar. Isso oferece uma segurança incidental (frequentemente confundida com um firewall de verdade). NAT não é um controle de segurança, mas dificulta ataques diretos de entrada.

③ Palavras-chave
NATPATprivate IPpublic IPport forwardingIPv4 exhaustionCGN

⑤ Como Funciona
O roteador mantém uma tabela de tradução mapeando (IP interno, porta interna) ↔ (IP externo, porta externa). Saída: reescreve IP/porta de origem e registra o mapeamento. Entrada: busca o IP/porta de destino na tabela e reescreve para o IP/porta interno. Sem um mapeamento existente, os pacotes de entrada são descartados (a menos que explicitamente encaminhados).
⑥ Ataque & Defesa
Ataque

  • UPnP — dispositivos abrem buracos no NAT automaticamente (frequentemente perigoso)
Defesa
  • Desative UPnP nos roteadores — a maioria dos usuários não precisa
  • Use um firewall de verdade — NAT não é uma barreira de segurança
  • Filtragem de egresso — controle o que pode sair da rede
⑦ Lembre-se
NAT = tradução, não proteção. A "segurança" do NAT é incidental. Sempre use um firewall de verdade — nunca confie no NAT para manter invasores fora.

Subnet
Dividindo redes em partes menores e gerenciáveis.

tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Uma subnet é um pedaço do espaço de IP reservado para um grupo específico."
Uma subnet (sub-rede) é uma divisão lógica de uma rede maior. Dentro de uma subnet, os dispositivos podem se comunicar diretamente sem passar por um roteador. Subnets são definidas pela notação CIDR como 192.168.1.0/24, onde /24 significa que os primeiros 24 bits identificam a rede e os 8 bits restantes identificam os hosts (256 endereços).
② Explicação

Subnetting é fundamental para segurança. A segmentação de rede divide a infraestrutura em zonas (servidores web, servidores de banco de dados, dev/staging/prod) para que uma violação em uma zona não alcance automaticamente as outras. Sem subnetting, uma rede plana significa que o movimento lateral é trivial assim que um invasor entra.

Matemática CIDR: /24 = 256 endereços, /16 = 65.536 endereços, /8 = 16M endereços. Números maiores significam subnets menores. Padrões comuns: redes domésticas /24, departamentos corporativos /22, grandes empresas dividem em muitas subnets conectadas por roteadores.

⑤ Como Funciona

O tráfego de broadcast fica dentro da subnet.

Ataque
  • Redes planas = comprometimento total a partir de qualquer ponto de violação — o invasor alcança tudo
  • Limites de subnet mal configurados — regras excessivamente permissivas entre zonas
  • ARP spoofing dentro de uma subnet para interceptar tráfego
Defesa
  • Microssegmentação — muitas subnets pequenas com regras rígidas entre elas
  • Network Access Control (NAC) para postura de endpoint
  • Arquitetura zero trust — verificar cada requisição entre subnets
⑦ Lembre-se
Subnetting = controle de danos. Quando (não "se") algo for comprometido, um bom subnetting limita até onde o estrago se propaga.

DHCP
Atribuição automática de endereço IP.
N·17DHCPauto-config
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"DHCP é como dispositivos obtêm um endereço IP sem você digitar um."
DHCP (Dynamic Host Configuration Protocol) atribui automaticamente endereços IP, máscaras de subnet, gateways padrão e servidores DNS a dispositivos que entram em uma rede. Quando seu celular se conecta ao WiFi, o DHCP fornece tudo o que ele precisa — instantaneamente.
② Explicação

DHCP funciona por meio de um processo de quatro etapas chamado "DORA": Discover (cliente faz broadcast procurando um DHCP), Offer (servidor oferece um IP), Request (cliente solicita formalmente), Acknowledge (servidor confirma e atribui). O cliente recebe um lease — um direito temporário de usar o IP por algum tempo.

Sem DHCP, todo dispositivo precisaria de configuração manual. Com DHCP, dispositivos entram e saem das redes sem complicação. Mas DHCP também é um vetor de ataque atraente — controlar o DHCP significa controlar a configuração de rede de todos que se conectam.

③ Palavras-chave
DHCPDORAleasebroadcastgatewaystatic IPreservation
④ Onde Aparece
Redes WiFi domésticas e de escritório
Redes cabeadas corporativas
WiFi de conferências e eventos
Redes celulares (processo similar)
⑤ Como Funciona
Quando um dispositivo entra na rede, ele faz broadcast de um DISCOVER. Qualquer servidor DHCP na rede pode responder com um OFFER. O cliente escolhe um e envia um REQUEST para o IP oferecido. O servidor confirma com um ACK e registra o lease (com prazo). O cliente começa a usar o IP. Antes do lease expirar, o cliente o renova. Se o dispositivo sair, o IP eventualmente fica disponível novamente.
⑥ Ataque & Defesa
Ataque
  • Rogue DHCP server — instalar um DHCP não autorizado que distribui DNS ou gateway malicioso, redirecionando o tráfego
  • DHCP starvation — esgotar o pool solicitando todos os IPs disponíveis, negando serviço a usuários legítimos
  • Sniffing do tráfego DHCP para mapear a rede
Defesa
  • DHCP snooping nos switches — permitir apenas respostas DHCP de servidores autorizados
  • Autenticação e isolamento de DHCP
  • Monitorar servidores DHCP não autorizados na rede
⑦ Lembre-se
DHCP é invisível até que algo dê errado. Servidores DHCP desonestos são sorrateiros e eficazes — eles redirecionam o tráfego antes que qualquer segurança de camada de aplicação entre em ação.

Kernel
O núcleo que fala com o hardware.
L·03kernelcore
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"O kernel é a camada mais baixa do OS — tudo o mais fica por cima."
O kernel é o programa central que gerencia recursos de hardware e fornece serviços a programas de usuário. Quando você executa um programa, ele na verdade pede ao kernel para fazer tudo que envolve hardware — ler arquivos, enviar pacotes, alocar memória.
② Explicação

Programas de usuário não podem falar com o hardware diretamente — eles fazem "chamadas de sistema" (syscalls) ao kernel. O kernel decide o que é permitido, agenda processos e gerencia memória. Essa separação é o que mantém o sistema estável: um app travado não trava o kernel.

O kernel Linux é monolítico — drivers rodam dentro dele para ter desempenho. Isso significa que um driver com bug pode travar tudo, mas a performance é excelente. Módulos carregáveis permitem adicionar funcionalidade sem reiniciar.

③ Palavras-chave

④ Onde Aparece
/proc/version mostra seu kernel
uname -a output
Bancos de dados de CVE do kernel
Carregamento de driver e erros de módulo do kernel
⑤ Como Funciona
Quando um processo precisa de qualquer coisa do sistema (abrir arquivo, enviar pacote, alocar RAM), ele dispara uma syscall. A CPU alterna de userspace para kernelspace, o kernel trata a requisição, retorna o resultado e volta. Centenas de syscalls acontecem por segundo por processo. O kernel usa algoritmos de escalonamento para compartilhar a CPU entre processos.
⑥ Ataque & Defesa
Ataque
  • Exploits de kernel — vulnerabilidades em syscalls ou drivers dão root ao atacante
  • Rootkits de kernel carregáveis se ocultam como módulos do kernel
  • Kernels desatualizados têm bugs conhecidos de escalonamento de privilégios (Dirty Pipe, Dirty COW)
Defesa
  • Mantenha o kernel atualizado — patches de segurança são críticos
  • Restrinja o carregamento de módulos do kernel em sistemas em produção
  • Use hardening de kernel (SELinux, AppArmor)
⑦ Lembre-se
Exploits de kernel são o Santo Graal — eles bypassam toda segurança de userspace. Sempre faça patch rápido quando CVEs de kernel aparecem.

"O sistema de arquivos do Linux é uma grande árvore que começa em / (root)."

⑤ Como Funciona

Defesa
  • Permissões rígidas em caminhos sensíveis — /etc/shadow deve ser 600 root:root
  • Monte /tmp com noexec para evitar execução a partir daí
  • Audite /etc e /var regularmente para alterações não autorizadas
⑦ Lembre-se
No Linux tudo é um arquivo, e todo arquivo tem um caminho começando em /. Domine a árvore e você domina o sistema.

Permissions
rwx para owner, group e others.
L·07permissionsrwx
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Permissões Linux: quem pode ler, escrever ou executar cada arquivo."
Permissões no Linux controlam acesso a arquivos via três classes (owner, group, others) e três ações (read, write, execute). Exibidas como -rwxr-xr-- no ls -l. Forma numérica: 754 (cada dígito é um valor de 3 bits).
② Explicação

Lendo rwxr-xr--: owner pode ler+escrever+executar, group pode ler+executar, others só podem ler. Numérico: r=4, w=2, x=1. Então 7=rwx, 5=r-x, 4=r--. Memorize isso e você lê qualquer número de chmod de relance.

Bits especiais importam para segurança: SUID (4xxx) faz um arquivo rodar como seu owner independente de quem o executa. SGID (2xxx) similar mas para group. Sticky bit (1xxx) em /tmp impede usuários de apagar arquivos uns dos outros. Arquivos SUID de propriedade do root são o caminho #1 para escalonamento de privilégios em CTFs.

③ Palavras-chave

④ Onde Aparece
ls -l output everywhere
chmod commands
Enumeração para escalonamento de privilégios (find / -perm -4000)

⑤ Como Funciona
Cada arquivo tem UID de owner, GID de group e uma palavra de permissão de 12 bits. Quando você tenta acessar um arquivo, o kernel verifica: você é o owner? Usa permissões do owner. Está no group? Usa permissões do group. Caso contrário usa permissões dos others. O bit SUID eleva o processo em execução aos privilégios do owner do arquivo durante a execução.
⑥ Ataque & Defesa
Ataque
  • Binários SUID-root com vulnerabilidades = shell root (verifique GTFOBins)
  • Arquivos com escrita pública no PATH permitem ao atacante substituí-los
  • Diretórios compartilhados mal configurados permitem usuários manipularem dados uns dos outros
Defesa
  • Princípio do least privilege — nunca dê chmod 777 em nada
  • Audite arquivos SUID regularmente: find / -perm -4000 -ls
  • Use ACLs para controle mais granular além do rwx
⑦ Lembre-se
rwx = 4+2+1. Memorize os números. SUID de propriedade do root é o que todo pentest Linux verifica primeiro.

User & Group
Identidade no Linux: UID e GID.
L·08usersgroups
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Todo processo e arquivo tem um usuário (UID) e grupo (GID) associado."
No Linux, toda ação é realizada por um usuário, identificado por um UID numérico. O usuário 0 é especial — é o root, com poder ilimitado. Usuários pertencem a um ou mais grupos, identificados por GIDs. As permissões verificam ambos.
② Explicação

Informações de usuário ficam em /etc/passwd (leitura pública — os hashes NÃO estão aqui apesar do nome). Hashes de senha ficam em /etc/shadow (só root). Definições de grupos em /etc/group. O grupo "wheel" ou "sudo" geralmente concede direitos de sudo.

Contas de serviço (www-data, nginx, mysql) têm UIDs baixos e shells limitados (/usr/sbin/nologin). Elas rodam serviços sem dar mais acesso do que o necessário. Um servidor web comprometido deve ter apenas privilégios de www-data, não root — isso é por design.

③ Palavras-chave

④ Onde Aparece
Comandos whoami e id

⑤ Como Funciona
Quando você faz login, o sistema cria uma sessão sob seu UID. Todo programa que você inicia herda esse UID e seus grupos. Verificações de acesso a arquivos comparam seu UID/GID com o owner/group do arquivo. Programas setuid mudam temporariamente para o UID do owner do arquivo durante a execução.
⑥ Ataque & Defesa
Ataque

  • Adicionar-se a grupos poderosos (docker = root via container escape)
  • Manipular sudoers para obter comandos root sem senha
Defesa
  • Políticas fortes de senha — bcrypt ou argon2 em /etc/shadow
  • Restrinja sudo com cuidado — nunca use NOPASSWD em comandos perigosos
  • Desative contas não utilizadas e remova UIDs órfãos
⑦ Lembre-se
UID 0 = root = modo deus. Todo escalonamento de privilégios é sobre ir de um UID baixo para UID 0.

tags: Estudado Revisar Difícil Favorito

"sudo permite que um usuário comum execute comandos específicos como root, com a própria senha."

② Explicação

Execute sudo -l para ver o que você pode rodar como root sem digitar a senha de root. Esse é um dos primeiros comandos que qualquer pentester executa após obter acesso ao shell. A saída revela seus potenciais caminhos para escalonamento.

Defesa
  • Sempre exija senha, a menos que absolutamente necessário
  • Limite regras sudo a comandos específicos necessários
  • Use caminhos absolutos completos nas entradas do sudoers
⑦ Lembre-se
sudo -l é seu mapa para o root. Sempre verifique. GTFOBins mostra como abusar de qualquer comando permitido pelo sudo.

  • Bloqueie as permissões dos arquivos de unit do systemd

⑤ Como Funciona
Ao rodar apt install nginx, o apt: consulta as listas de pacotes em /etc/apt/sources.list, encontra nginx e suas dependências, baixa os pacotes .deb, verifica assinaturas GPG, roda scripts pré-instalação, copia arquivos para os locais padrão, roda scripts pós-instalação, atualiza o banco de dados de pacotes. dpkg é a ferramenta de mais baixo nível que o apt usa por baixo.
⑥ Ataque & Defesa
Ataque
  • Pacotes desatualizados com CVEs conhecidos — primeira coisa a enumerar (dpkg -l, rpm -qa)
  • Repositórios comprometidos distribuem atualizações maliciosas (raro mas acontece)
  • Adicionar repositório malicioso para instalar pacotes backdoored
Defesa
  • apt update && apt upgrade regularmente — automatize quando possível
  • Verifique assinaturas GPG dos pacotes — nunca desabilite
  • Fixe versões de repositórios confiáveis e minimize repos de terceiros
⑦ Lembre-se
Atualize ou seja comprometido. A maioria dos servidores cai por CVEs conhecidos em pacotes sem patch — não zero-days. apt upgrade é seu melhor amigo.

Logs
Onde o Linux registra tudo que acontece.
L·12logs/var/log
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Logs são a linha do tempo do seu sistema — leia-os quando as coisas derem errado."
Logs são arquivos de texto onde o sistema, serviços e apps registram eventos. A maioria fica em /var/log/. Arquivos críticos: auth.log (logins, sudo), syslog (geral), kern.log (kernel), logs do Apache/nginx. Sistemas modernos usam journalctl também.
② Explicação

Quando algo quebra ou você suspeita de comprometimento, os logs são o ponto de partida. Tentativas falhas de SSH, abuso de sudo, erros de kernel, crashes de aplicação — tudo registrado. Ler logs é uma habilidade essencial tanto na administração quanto em forense.

Para atacantes, logs são evidências a remover. Limpar /var/log/auth.log após comprometimento é clássico. Para defensores, é por isso que logging centralizado (servidor syslog, servidor, SIEM) importa — cópias dos logs saem do host comprometido imediatamente. Uma vez que o atacante está dentro, você não pode confiar nos logs locais.

③ Palavras-chave
logs/var/logauth.logsyslogjournalctlkern.logrsysloglogrotate
④ Onde Aparece
Diagnóstico de qualquer falha
Investigações forenses
Ingestão em IDS/SIEM
Auditorias de conformidade
⑤ Como Funciona
Programas e o kernel escrevem eventos em arquivos de log via syslog ou systemd-journal. Logs rotacionam periodicamente — entradas antigas são comprimidas e arquivadas (logrotate). Em sistemas modernos, journalctl exibe os journals binários do systemd. Filtragem: tail -f para tempo real, grep para busca, awk para parsing.
⑥ Ataque & Defesa
Ataque
  • Limpeza de logs após comprometimento — apagar auth.log para esconder rastros
  • Injeção de log — controlar input que é registrado sem sanitização
  • Exploits de logrotate e race conditions em arquivos de log
Defesa
  • Centralize logs — envie para um servidor de log separado e hardened
  • Torne logs append-only quando possível (chattr +a)
  • Monitore quedas no volume de logs (silêncio repentino é suspeito)
⑦ Lembre-se
Logs respondem "o que aconteceu?" Leia-os com frequência. Centralize-os com dedicação. O atacante que apaga /var/log/auth.log no host comprometido nada pode fazer contra o SIEM que já recebeu as entradas.

SSH
Shell remoto seguro — a linha de vida de todo servidor.
L·13sshremote
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"SSH dá a você acesso shell criptografado a uma máquina remota."
SSH (Secure Shell) é o protocolo padrão para acesso remoto seguro. Conecta com criptografia a uma máquina remota e dá acesso shell. Escuta na TCP porta 22 por padrão. Substitui protocolos inseguros como telnet (tudo em cleartext).
② Explicação

SSH suporta autenticação por senha, por chave (muito mais segura) e por certificado. Pares de chave pública/privada são o padrão: mantenha a chave privada em segredo, coloque a chave pública em cada servidor que você quer acessar. Adicione uma passphrase à chave privada para proteção extra.

SSH também é um túnel. ssh -L permite encaminhar portas — útil para acessar serviços atrás de firewalls. Túneis SSH reversos (ssh -R) são como atacantes mantêm persistência após uma invasão. SSH é uma das ferramentas mais poderosas no arsenal de qualquer sysadmin e atacante.

③ Palavras-chave
SSHport 22private keypublic key~/.sshssh-keygentunnelauthorized_keys
④ Onde Aparece
Administração remota de servidores
Git via SSH
Persistência via chaves SSH
Tunneling e pivoting
⑤ Como Funciona
O cliente conecta ao servidor na porta 22. Ambos negociam algoritmos de criptografia. O servidor apresenta sua chave pública (verificada contra ~/.ssh/known_hosts). O cliente se autentica (senha ou chave). Uma vez autenticado, todo o tráfego flui pelo túnel criptografado — você tem um shell como usuário remoto. Chaves ficam em ~/.ssh/ em ambos os lados.
⑥ Ataque & Defesa
Ataque
  • Bruteforce de senhas SSH na porta 22 (sempre exposta em algum lugar)
  • Roubar chaves privadas SSH de laptop comprometido = acesso a todos os servidores que ele toca
  • Persistência via authorized_keys — inserir chave pública do atacante para acesso permanente
Defesa
  • Desative autenticação por senha — apenas chaves
  • Use passphrases fortes nas chaves
  • Mude SSH da porta 22 (segurança por obscuridade, mas reduz ruído)
  • Monitore authorized_keys para entradas não autorizadas
⑦ Lembre-se
Chaves SSH = seu reino. Perca-as e perde todos os servidores. Audite ~/.ssh/authorized_keys regularmente em todo host.

Cron
Agendamento de tarefas para rodar automaticamente.
L·14cronscheduling
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"Cron executa comandos em agendamento — backups, limpezas, qualquer coisa repetitiva."
cron é o agendador de tarefas padrão do Linux. Cada usuário tem um crontab (tabela cron) listando comandos agendados. O daemon cron os executa nos horários especificados. Formato: 5 campos de tempo + comando. 0 3 * * * significa "3h da manhã todo dia".
② Explicação

Cada usuário pode ter um crontab via crontab -e. O sistema também tem /etc/crontab e /etc/cron.d/, mais os diretórios /etc/cron.daily, /etc/cron.weekly, /etc/cron.monthly que executam scripts nessas frequências.

O cron está em todo lugar porque automação precisa de agendamento. É também um favorito de atacantes que buscam persistência. Um cron job que roda a cada 5 minutos como root, baixando e executando um payload, é um backdoor clássico. Revisar entradas do cron é essencial durante resposta a incidentes.

③ Palavras-chave
croncrontabscheduling/etc/crontabcron.dailypersistencesystemd timer
④ Onde Aparece
Backups, rotação de logs, manutenção do sistema
Caminhos de escalonamento de privilégios em CTF
Persistência após comprometimento
Sistemas modernos também usam systemd timers
⑤ Como Funciona
O cron daemon (crond) roda em segundo plano. A cada minuto ele verifica todas as crontabs (de usuário e do sistema). Se o horário de uma tarefa bate com o horário atual, o cron bifurca um processo e executa o comando como o usuário correto. A saída é enviada por email ao usuário (ou descartada se email não estiver configurado).
⑥ Ataque & Defesa
Ataque
  • Persistência — inserir payload do atacante em /etc/cron.d ou crontab do usuário
  • Cron jobs rodando como root com scripts graváveis = root imediato
  • Race conditions em scripts cron podem ser exploradas
Defesa
  • Audite cron jobs regularmente — liste todos por usuário
  • Restrinja acesso ao cron (cron.allow / cron.deny)
  • Torne os scripts de cron job somente leitura e de propriedade do root
⑦ Lembre-se
Cron jobs são o mecanismo de persistência favorito de atacantes. Após qualquer incidente, varra todas as crontabs — especialmente /etc/cron.* e a do root.

systemd
O sistema de init e gerenciador de serviços.
L·15systemdservices
tags: Estudado Revisar Difícil Favorito
① Definição Simples
"systemd inicia tudo quando você liga e gerencia serviços durante a execução."
systemd é o sistema de init moderno usado pela maioria das distros Linux. É o primeiro processo iniciado pelo kernel (PID 1) e inicia todo o resto. Também gerencia serviços, timers, mounts e mais. Substitui o SysV init mais antigo.
② Explicação

Serviços são definidos em unit files (/etc/systemd/system/*.service ou /lib/systemd/system/*.service). systemctl é a ferramenta: systemctl start nginx, systemctl status, systemctl enable. Os logs vão para journalctl.

systemd é controverso (principalmente porque faz muitas coisas) mas é a realidade no Ubuntu, Debian, Fedora, RHEL, Arch — praticamente em todo lugar. Conhecer systemctl e journalctl é essencial. Os timers do systemd também estão substituindo o cron em muitos casos — mais flexíveis e integrados com o logging.

③ Palavras-chave
systemdsystemctlunit fileservicejournalctl.serviceenablemask
④ Onde Aparece
Gerenciamento de serviços no Linux moderno
Verificações com systemctl status no trabalho de admin
journalctl para logs do sistema
Persistência via unidades de serviço maliciosas
⑤ Como Funciona
O systemd lê unit files que definem serviços. O systemctl fala com o daemon systemd para iniciar, parar e habilitar serviços. Serviços habilitados iniciam no boot. Cada serviço roda como seu próprio processo supervisionado pelo systemd — se travar, o systemd pode reiniciá-lo. O journalctl lê o journal binário do systemd para logs.
⑥ Ataque & Defesa
Ataque
  • Persistência — inserir arquivo .service malicioso com WantedBy=multi-user.target para persistência no boot
  • Serviços rodando como root com caminhos ExecStart graváveis podem ser sequestrados
  • Desabilitar serviços de segurança (auditd, falco) para operar sem ser detectado
Defesa
  • Audite /etc/systemd/system/ regularmente para unidades desconhecidas
  • Restrinja as permissões dos unit files — somente root pode escrever
  • Use diretivas de segurança do systemd (PrivateTmp, NoNewPrivileges) para serviços
⑦ Lembre-se
systemctl + journalctl é o kit do admin moderno. Domine os dois. Unidades systemd maliciosas são os novos cron jobs para persistência.
